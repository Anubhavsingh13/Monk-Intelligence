# Claude Code: Modes, Instructions & Writing Prompts That Actually Work


**Last Updated:** 2026-04-15

> *The same task, written two different ways, can produce a careful plan or a broken deployment. The words you choose are the architecture decisions of an agentic session.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Explain the difference between Plan, Auto, and Accept Edits modes and choose the right one for a given task
- Identify the signals in a task that tell you which mode to reach for before you start
- Write a prompt that specifies functional requirements, non-functional constraints, and target environment clearly enough that Claude Code produces usable output on the first attempt
- Rewrite a vague instruction into a high-quality one — and explain why the second version works better
- Craft instructions for Azure-specific tasks (infrastructure, deployment, configuration) that surface the right assumptions and constraints upfront
- Recognize when a running session has gone off-track and know how to course-correct without starting over

---

## 1. Why This Matters (In Our Systems)

Two engineers. Same task: *"add retry logic to our Azure Service Bus consumer."*

Engineer A types exactly that and hits enter in Auto mode. Claude Code writes retry logic — clean, generic, runs locally. It uses an in-memory backoff library. It doesn't know your team uses `azure-service-bus` SDK v7, that you have a dead-letter queue pattern already in place, or that the non-functional requirement is "no more than 3 retries with exponential backoff, no retry on `ServiceBusError` with reason `MessageLockLost`." The code passes review because it looks reasonable. It goes to staging. Three days later, a cascading retry storm takes down the consumer service.

Engineer B takes 90 seconds to write a proper instruction. Claude Code produces the right implementation, with the right SDK method, respecting the existing dead-letter pattern.

The difference wasn't the model. It was the instruction quality and the mode choice.

The engineers who learn this skill stop being surprised by what Claude Code produces and start being in control of it.

## 2. Intuition & Mental Models

Think about how you'd brief a contractor you've just hired. You wouldn't say "build me a house" and walk away. You'd specify: what it's for, how many rooms, what materials are preferred, what the budget ceiling is, what the site constraints are, and what you absolutely cannot compromise on.

Claude Code is a fast, capable contractor. The modes are the *engagement style* — how much oversight you want to maintain:

- **Plan mode** is you saying *"before you touch anything, show me your blueprint and wait for sign-off."* The contractor draws plans, explains trade-offs, and doesn't pick up a hammer until you say go.
- **Auto mode** is *"I trust you, here are the specs, get it done."* The contractor builds. You review at the end.
- **Accept Edits mode** is the middle ground: *"build, but check with me before doing anything that can't be undone."* The contractor moves fast but pauses at each significant decision.

The mode doesn't change the contractor's competence. It changes your visibility and control over the process.

**Prompting intuition:** A prompt is not a search query. It's a brief. Good briefs answer four questions even when not asked:
1. What is the goal?
2. What are the constraints?
3. What is the environment?
4. What does success look like?

Every word in your prompt that answers one of these is a word Claude Code doesn't have to guess at.

## 3. Core Concepts & Terminology

**Plan mode** — Claude Code reads the codebase, reasons about the task, and produces a written plan of what it intends to do. It does not write or edit any files until you explicitly approve the plan. Ideal for high-risk or high-complexity tasks.

**Auto mode** — Claude Code executes the full task loop autonomously: reading, planning, writing, running commands, checking results. It may ask for clarification at decision points but generally proceeds without waiting for approval at each step.

**Accept Edits mode** — Claude Code executes autonomously but pauses and shows you a diff before applying each file write. You approve or reject each edit individually. Shell commands still require confirmation. This is the default for many users.

**Functional requirement** — What the code must *do*. "Retry failed Service Bus messages up to 3 times."

**Non-functional requirement (NFR)** — How the code must *behave* under constraints. "Each retry must use exponential backoff with jitter. Total retry window must not exceed 30 seconds. Must not retry on `MessageLockLost` errors."

**Environment context** — The specific platform, SDK version, infrastructure, and deployment model your code runs in. "Azure Functions v4 runtime, Node.js 20, Service Bus SDK v7, deployed via GitHub Actions to West Europe."

**Instruction precision** — The degree to which a prompt removes ambiguity about goal, constraints, environment, and success criteria. Higher precision correlates directly with higher first-attempt output quality.

## 4. How It Works (What Actually Matters)

### The Three Modes — Mechanics

**Plan mode** is activated with:

```bash
claude --plan
```

Or within a session by prefixing your message with `/plan`. Claude Code will:
1. Read all files it deems relevant
2. Output a numbered plan: what it will change, why, and what it will leave alone
3. Stop and wait

You respond with approval, amendments, or rejection. Only on explicit approval does it proceed.

**Auto mode** is activated with:

```bash
claude --dangerously-skip-permissions
```

Or by setting permissions to allow all in `settings.json`. Claude Code runs the full agent loop without pausing. You see output streaming in real time and can interrupt with `Ctrl+C` at any point.

> ⚠️ **Counterintuitive:** Auto mode is not "reckless mode." It is appropriate for well-specified tasks in trusted environments. The risk isn't the mode — it's pairing Auto mode with a vague instruction. A precise instruction in Auto mode is often safer than a vague instruction in Plan mode, because the plan itself will be based on wrong assumptions.

**Accept Edits mode** is the default interactive behaviour when you run `claude` without flags. Each file write surfaces a diff:

```
Edit to src/servicebus/consumer.ts:

  + import { exponentialBackoff } from './retry';
  ...
  - await receiver.receiveMessages(10);
  + await receiver.receiveMessages(10, { retryOptions: exponentialBackoff });

Apply this edit? [y/n/a(ll)/s(kip)]
```

You have four choices: yes, no, yes-to-all-remaining, skip-this-one.

### How Instruction Quality Shapes Output

Claude Code processes your instruction as its planning input. Every ambiguity in the instruction becomes a decision point where it must guess. Here is what it guesses from:

1. What it can infer from the codebase it has read
2. Common patterns in training data for similar tasks
3. The most recently mentioned context in the conversation

When the codebase provides clear signals (existing patterns, typed interfaces, named conventions), guessing works reasonably well. When it doesn't — new infrastructure, unfamiliar SDK, Azure-specific behaviour — guessing produces generic output that is plausible but wrong.

> ⚠️ **Counterintuitive:** Adding more words to a prompt only helps if those words remove ambiguity. Writing "please add retry logic carefully and make sure it works well" is longer but provides zero additional constraint. Writing "retry on `ServiceBusError` only when `error.code === 'GeneralError'`" is shorter and provides exact constraint. Precision beats verbosity.

## 5. Worked Examples & Realistic Scenarios

### Example 1: Choosing the Right Mode

**Task:** Migrate an existing Express app from CommonJS to ESM.

This is high-risk (many files, structural change, easy to break imports silently), not fully reversible if you don't have clean git state, and benefits from a review step before anything is written.

**Correct mode: Plan**

```bash
claude --plan
```

```
Migrate this Express app from CommonJS (require/module.exports) to ESM
(import/export). The app runs on Node.js 20. Start by listing every file
that will change and what specifically needs to change in each one. Do not
make any edits until I approve the plan.
```

You review the plan. You catch that it missed a dynamic `require()` in a config loader that can't be converted naively. You flag it. The plan is amended. Then you approve.

---

**Task:** Add a health check endpoint to an Azure Functions app.

This is low-risk, isolated, easy to verify, and you've done it ten times before.

**Correct mode: Auto (or Accept Edits)**

```
Add a health check HTTP trigger to this Azure Functions v4 app (Node.js 20).
The endpoint should be GET /health, return HTTP 200 with JSON body
{ "status": "ok", "timestamp": "<ISO8601>" }, require no auth
(authLevel: anonymous), and follow the existing function file structure in src/functions/.
```

Auto mode handles this cleanly. The task is well-specified, the environment is explicit, and the success criteria are unambiguous.

---

### Example 2: Vague vs. Precise Instructions (Azure)

**Vague:**
```
Add logging to the Azure function
```

Claude Code will add *some* logging. Probably `console.log`. Maybe Application Insights if it spots the SDK in `package.json`. Maybe not. The log level, structured vs. unstructured format, correlation ID, and whether it integrates with your existing telemetry pipeline — all guessed.

**Precise:**
```
Add structured logging to the Azure Function in src/functions/processOrder.ts.

Requirements:
- Use the @azure/logger package already installed in this project
- Log at INFO level on function start with: { trigger: "servicebus", messageId, correlationId }
- Log at ERROR level on any thrown exception with: { error.message, error.stack, messageId }
- Do not add console.log — we pipe logs to Application Insights via the existing
  appInsights client in src/telemetry/client.ts
- Non-functional: logging must not throw — wrap in try/catch if needed
```

The second version gives Claude Code: the target file, the existing SDK to use, the exact log shapes, the existing integration point, and the NFR (logging must not throw). The output will be usable on the first attempt.

---

### Example 3: Non-Functional Requirements for Azure Infrastructure

NFRs are the most commonly omitted part of instructions. Here's what they look like in practice:

```
Provision an Azure Storage account for storing user-uploaded documents.

Functional:
- Blob storage, container named "user-documents"
- SAS token generation for upload URLs (write-only, 15-minute expiry)

Non-functional:
- Region: West Europe (to match existing resource group rg-prod-westeu)
- Redundancy: ZRS (Zone-Redundant Storage) — not LRS
- Access tier: Hot
- Network: restrict public access; allow only from VNet subnet snet-app-prod
- Encryption: customer-managed keys via existing Key Vault kv-prod-westeu
- Naming convention: sa{environment}{region_short}{purpose} e.g. saprodweuuserdocs

Use Bicep, not ARM JSON. Output a single .bicep file at infra/storage/userDocs.bicep.
```

Without the NFR block, Claude Code would provision a storage account — but probably LRS, public access, Microsoft-managed keys, and a generated name. All technically correct. All wrong for production.

## 6. Practical Usage & Decision Guidance

### Mode Selection Decision Table

| Task Characteristics | Recommended Mode |
|---|---|
| Many files change, hard to reverse | Plan |
| New to this codebase, uncertain scope | Plan |
| Infrastructure or deployment changes | Plan |
| Security-sensitive (auth, secrets, IAM) | Plan |
| Well-specified, single file, low risk | Auto or Accept Edits |
| Repetitive task you've done before | Auto |
| You want to review each file change | Accept Edits |
| Automated pipeline (CI, scripts) | Auto with precise instruction |
| Exploratory — you're not sure what you want | Accept Edits + iterate |

### Instruction Quality Checklist

Before submitting an instruction, ask:

- [ ] Have I named the specific file(s) or module(s) involved?
- [ ] Have I stated the platform/SDK/runtime version?
- [ ] Have I separated functional requirements from non-functional requirements?
- [ ] Have I named any existing patterns, helpers, or conventions to follow?
- [ ] Have I stated what success looks like (return value, log output, test pass)?
- [ ] Have I named anything that must *not* change or be touched?

You don't need all six on every task. A one-line fix doesn't need an NFR block. But for anything that touches infrastructure, deployment, auth, or multiple files — every missing checkbox is a guess Claude Code has to make.

## 7. Common Pitfalls & Misconceptions

**"Plan mode is always safer."**
Most people assume reviewing a plan before execution eliminates risk. The reality: a plan built on a vague instruction is a plan full of assumptions. You're reviewing the plan, not the assumptions beneath it. A bad instruction in Plan mode produces a confident-sounding plan that is wrong. Precision of instruction matters more than mode choice.

**"Auto mode means I don't need to pay attention."**
Auto mode removes the confirmation step, not your responsibility. You should still watch the output stream in real time. The `Ctrl+C` interrupt is always available. Treating Auto mode as "walk away and come back" is how you end up with 47 files changed that you have to untangle.

**"More detail always helps."**
The reality: irrelevant detail dilutes signal. If you paste your entire architecture document before asking Claude Code to fix a typo in a config file, the relevant context drowns in noise. Write instructions proportional to the task's complexity and risk.

**"I'll fix the NFRs in review."**
NFRs that aren't in the instruction often don't surface in the output at all — Claude Code doesn't know what it didn't know to ask about. "It should be secure" in review is too late if the implementation chose the wrong auth pattern four decisions ago. State NFRs upfront or plan for a rewrite.

**"The mode is set globally and that's fine."**
A global Auto mode setting on your personal machine is fine. The same setting checked into `.claude/settings.json` in a shared repo means every engineer who clones it gets Auto mode by default. Scope your permissive settings to your local environment, not the repo.

## 8. Trade-offs, Scale, and Edge Cases

Plan mode adds a round-trip: Claude Code reads, plans, waits, then executes. For short tasks, this doubles the time for marginal safety gain. For long tasks, it's worth every second — catching a wrong assumption in a 5-line plan is far cheaper than catching it in a 300-line diff.

Accept Edits mode works beautifully for medium-complexity tasks but becomes friction-heavy when a task legitimately changes 20+ files (e.g., a large refactor). In those cases, review the plan once in Plan mode and then switch to Auto for execution. You get strategic oversight without per-file friction.

For Azure specifically: Claude Code's training includes general Azure SDK patterns, but it may lag on features released after its knowledge cutoff, or on opinionated conventions your team has adopted. Always name the SDK package and version. If you're using preview APIs or custom wrappers, describe them explicitly — Claude Code won't infer them from a package name it hasn't seen.

In CI pipelines, Auto mode with a highly constrained instruction is the right pattern. The instruction does the work of the human reviewer. A pipeline that runs `claude` with a vague instruction in Auto mode is a pipeline that will eventually merge something surprising.

3. You're in Accept Edits mode and Claude Code shows you a diff you don't understand. What are your options, and which is better than blindly approving?

5. You've been using Plan mode for a task. The plan looks correct. You approve it and Claude Code starts executing — then it does something unexpected. What likely went wrong, and what would you do differently next time?

## Summary

Claude Code's three modes — Plan, Auto, and Accept Edits — are not a quality dial from "careful" to "reckless." They are visibility controls, and the right one depends on the risk profile of the task. A vague instruction in Plan mode is still a vague instruction; the mode does not fix the brief. Instruction quality — separating functional requirements from non-functional ones, naming the environment explicitly, and stating what must not change — is the single biggest lever you have over output quality. For Azure work specifically, environment context (SDK version, region, naming conventions, network constraints) is almost never inferrable and almost always missing from first-attempt instructions.
