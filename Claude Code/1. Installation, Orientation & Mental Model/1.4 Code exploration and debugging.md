# Claude Code: Codebase Exploration & Bug Fixing Workflows

**Last Updated:** 2026-04-15

> *The fastest way to understand a codebase you've never touched — or fix a bug you can't reproduce — is to stop reading alone and start asking the right questions out loud.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Use Claude Code to generate a navigable map of an unfamiliar codebase — architecture, data flow, module ownership
- Ask focused questions that produce explanation at the right depth — not too shallow, not overwhelming
- Submit a technical bug (stack trace, error log) to Claude Code and guide it to a root cause
- Submit a functional bug (wrong behaviour, no error) and structure the investigation so Claude Code can trace logic rather than guess
- Distinguish between symptoms and root causes — and prompt Claude Code at the right level
- Validate a proposed fix before accepting it — and know what "good enough" looks like

---

## 1. Why This Matters (In Our Systems)

Every engineer has had this week: you're handed a bug in a service you've never touched. The original author is unavailable. The docs are six months stale. The stack trace points into a module with four layers of abstraction, and you have three hours before the fix needs to be in staging.

The old workflow: read the module top to bottom, grep for the symbol, trace through call sites, read again, form a hypothesis, test it, repeat. Two to four hours for a confident mental model.

The new workflow: open Claude Code, describe what you see, let it trace. The model can hold the call graph in context, follow the imports you can't see at a glance, and explain the interaction that caused the symptom — in minutes.

But there's a failure mode. Engineers paste a stack trace, get a plausible-sounding answer, apply the fix, and ship. The fix addressed the symptom. The root cause is still live. Two weeks later, a different codepath triggers the same underlying issue.

This page teaches both skills: how to get a real map, and how to push past the first plausible answer to the actual root cause.

## 2. Intuition & Mental Models

### For Codebase Exploration

Think of Claude Code as a search light, not a floodlight. You don't turn it on and illuminate everything at once — you point it. The quality of your map depends on the quality of your questions.

A good exploration session works like an interview with someone who's read every file: you start with the big picture ("what does this system do?"), move to the structure ("how is it organised?"), then zoom into the parts you'll actually touch ("walk me through this specific flow").

A bad session asks everything at once and gets a summary that feels complete but misses the one module where the interesting behaviour lives.

### For Bug Fixing

Here's the mental model that matters: **a bug has a symptom, a trigger, and a root cause. They are almost never the same thing.**

- **Symptom:** what you observe. *"The order confirmation email isn't sent."*
- **Trigger:** the condition that surfaces the bug. *"Only happens when the order contains a digital-only product."*
- **Root cause:** the actual code defect. *"The email dispatch is gated on `order.hasPhysicalItems`, which is false for digital orders — an assumption baked in six months ago that no longer holds."*

Claude Code is excellent at tracing from symptom to trigger to root cause — if you give it all three layers to work with. If you only give it the symptom, it will find the nearest plausible fix for the symptom. That's not the same thing.

> The best bug fixing sessions read like a detective story: you present the evidence, Claude Code forms hypotheses, you validate or eliminate each one, and you converge on the actual defect together.

## 3. Core Concepts & Terminology

**Codebase map** — A mental (or written) model of how a system is structured: what the main modules are, what each one owns, how data flows between them, and where the key decision points live. Claude Code can produce this as a written artifact you keep in `CLAUDE.md` or as an interactive explanation.

**Call graph** — The chain of function calls triggered by a given action. *"User hits POST /checkout → `checkoutController` → `OrderService.createOrder` → `PaymentGateway.charge` → `EmailDispatcher.sendConfirmation`."* Tracing the call graph is how you find where a bug lives in code you've never read.

**Symptom vs. root cause** — The observable failure vs. the actual defect. Fixing the symptom without the root cause is the most common source of recurring bugs.

**Hypothesis-driven debugging** — Forming an explicit guess about the root cause, then testing it, rather than reading code until something looks wrong. Claude Code works best when you give it a hypothesis to validate or disprove.

**Functional bug** — A bug with no error or exception. The code runs successfully, but the output is wrong. These are harder to trace because there's no stack trace — only behaviour to compare against expectation.

**Technical bug** — A bug that produces an explicit error: an exception, a failed assertion, a non-2xx response with a logged cause. These have a direct evidence trail.

## 4. How It Works (What Actually Matters)

### Exploration: How Claude Code Builds a Map

When you ask Claude Code to explore a codebase, it uses the same on-demand reading model described in earlier pages: it starts at the entry point you give it, reads that file, follows imports and references, and builds a picture of the module graph from the inside out.

This means:

- **What you point it at shapes the map.** Start from `index.ts` or `app.py` and you get the application-level view. Start from `src/payments/` and you get the payments subsystem view. Both are valid — the right one depends on what you need to understand.
- **It reads code literally.** Comments, dead code, and stale abstractions will appear in the map unless they're clearly marked. A well-maintained codebase produces a cleaner map.
- **It can miss implicit coupling.** If module A affects module B through a shared database table or a message queue (not a direct import), Claude Code won't see that connection unless you tell it to look.

> ⚠️ **Counterintuitive:** Asking "explain this entire codebase" produces worse output than "explain how a user order flows through this codebase from API request to database write." The second question gives Claude Code a thread to follow. The first gives it nothing to anchor on.

### Bug Fixing: How Claude Code Traces a Bug

For technical bugs, Claude Code follows the stack trace as an entry point: it reads the files named in the trace, follows the call chain backward toward the source, and forms a hypothesis about which assumption was violated.

For functional bugs, it has no stack trace. You become the evidence. The quality of your bug description — what happened, what was expected, what the trigger condition was — determines whether Claude Code finds the right module or spends turns reading unrelated code.

> ⚠️ **Counterintuitive:** Claude Code will almost always produce a plausible first answer. "Plausible" is not the same as "correct." The most valuable skill in a bug-fixing session is knowing how to push past the first answer — asking Claude Code to explain *why* the fix works and what else could cause the same symptom. If it can't explain the mechanism, the fix is a guess.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Mapping an Unfamiliar Codebase

You've joined a team. There's a Node.js backend you've never touched, no architecture doc, and a bug assigned to you in the order processing module.

**Step 1 — Get the big picture:**

```
I'm new to this codebase. Starting from src/index.ts, give me a 
high-level map of this system: what are the main modules, what does 
each one own, and how does a request flow from the API layer to 
the database? Focus on the happy path for a POST /orders request.
```

Claude Code reads `index.ts`, follows the router, finds the controller, traces into the service layer, finds the repository. It produces a written flow — not exhaustive, but enough to orient you.

**Step 2 — Drill into the relevant module:**

```
Focus on src/orders/OrderService.ts. Walk me through what createOrder() 
does step by step. What are its dependencies? What can it throw? What 
does it assume about the input it receives?
```

The second question is targeted. You're not asking it to re-read the whole system — you're asking it to go deep on the one file that matters for your bug.

**Step 3 — Surface the implicit:**

```
Does OrderService have any dependencies on external services or queues 
that aren't obvious from reading the file? Check its imports and look 
for any pub/sub or event emission that might trigger side effects.
```

This is the question most engineers forget. The answer is where the bug often lives.

---

### Scenario 2: Technical Bug — Stack Trace Investigation

**The error (from your Azure Application Insights logs):**

```
TypeError: Cannot read properties of undefined (reading 'customerId')
    at OrderService.createOrder (src/orders/OrderService.ts:47:32)
    at async OrderController.post (src/orders/OrderController.ts:23:18)
```

**Your prompt:**

```
I'm seeing this error in production on our Azure Function:

TypeError: Cannot read properties of undefined (reading 'customerId')
    at OrderService.createOrder (src/orders/OrderService.ts:47:32)
    at async OrderController.post (src/orders/OrderController.ts:23:18)

Start from OrderController.ts line 23. Trace what gets passed to 
createOrder(), where it comes from, and what condition would make it 
undefined at line 47. Don't propose a fix yet — I want to understand 
the root cause first.
```

Why "don't propose a fix yet" matters: it forces Claude Code to fully trace the call path before jumping to a solution. Without it, the first response is often `if (!data?.customerId) return` — a defensive guard that masks the upstream problem rather than fixing it.

Claude Code reads `OrderController.ts:23`, reads what `createOrder` receives, traces back to the request parsing — and finds that the controller extracts `customerId` from `req.body.customer.id` but some order types send `req.body.customerId` directly (a schema inconsistency introduced three months ago).

**Follow-up:**

```
Now propose a fix. The fix should address the schema inconsistency, 
not just add a null guard. Include what the downstream impact of 
normalising the schema is — are there other callers of createOrder 
that might be affected?
```

---

### Scenario 3: Functional Bug — Wrong Behaviour, No Error

**The symptom:** Users with a "premium" subscription are not seeing the discounted price on checkout. No errors in logs. The discount logic exists and works in unit tests.

**Your prompt:**

```
There's a functional bug: premium subscribers are not getting the 
discounted price at checkout, but no errors are thrown. The unit 
tests for discount logic pass.

Relevant files I know of:
- src/pricing/DiscountService.ts — applies discounts
- src/checkout/CheckoutService.ts — orchestrates checkout
- src/users/SubscriptionService.ts — manages subscription state

Start by reading CheckoutService.ts and trace how it determines 
whether a discount should be applied. I want to understand the 
decision path, not jump straight to a fix.
```

Claude Code reads `CheckoutService.ts`, finds the discount check, reads the condition — and discovers that `isEligibleForDiscount()` calls `SubscriptionService.getStatus()` which returns `"PREMIUM"` for active premium users but the comparison in `DiscountService.ts` checks for `"premium"` (lowercase). Case mismatch. Unit tests passed because they used the same lowercase string in both the mock and the assertion.

**The lesson embedded in this scenario:** functional bugs often live at the boundary between two modules. The individual unit is correct. The integration assumption is wrong. Asking Claude Code to trace the *decision path* — not just the logic of one function — is how you find these.

## 6. Practical Usage & Decision Guidance

### Exploration Prompts by Goal

| You want to understand... | Ask Claude Code to... |
|---|---|
| Overall system structure | Trace the main request/event flow from entry point to persistence |
| A specific module | Read the module and explain: what it owns, what it assumes, what can go wrong |
| Data flow | Follow a specific piece of data (e.g., `order`) from creation to storage |
| Hidden coupling | Look for events, queues, shared state, or database side effects not visible in imports |
| Historical decisions | Read the relevant files and note any patterns that look like past constraints |

### Bug Fixing Approach by Bug Type

| Bug type | Starting point | Key question to ask |
|---|---|---|
| Stack trace (technical) | The file and line in the trace | "What condition produces this state at this line?" |
| Wrong output (functional) | The module that produces the output | "Trace the decision path — where is the output value determined?" |
| Works locally, fails in Azure | Environment and config files | "What environment-specific values or service bindings differ?" |
| Regression (worked before) | Git history + the affected module | "What changed recently in this module or its dependencies?" |
| Intermittent / race condition | Async boundaries and shared state | "Where is shared state written and read without synchronisation?" |

### When Not to Use Claude Code for Debugging

If the bug requires live production data you can't share, or depends on runtime state that only exists in a specific Azure environment, Claude Code can help you form hypotheses but cannot trace execution. In these cases, use Claude Code to write the diagnostic instrumentation — then run it yourself and bring the output back.

## 7. Common Pitfalls & Misconceptions

**"I gave it the error and it gave me a fix — done."**
Most people assume the first answer is the answer. The reality: Claude Code is optimising for a plausible response, not necessarily the root cause. If you can't articulate *why* the fix works — not just *what* it does — you haven't found the root cause. Always ask Claude Code to explain the mechanism.

**"I asked it to explain the whole codebase."**
A request with no anchor produces a response with no depth. Claude Code will produce a high-level summary that sounds comprehensive but glosses over the exact module where the interesting behaviour lives. Give it a thread to follow — a specific request type, a specific data entity, a specific feature.

**"The unit tests pass so the logic is correct."**
Unit tests test a unit in isolation. Functional bugs — especially in integration-heavy systems — live at the boundary between units. If tests pass and the behaviour is wrong, the first question to ask is: "what assumption does the test make that the real integration doesn't satisfy?"

**"I described the symptom but not the trigger."**
Claude Code needs the trigger condition to narrow the search space. "Users aren't seeing discounts" covers the whole discount system. "Premium users with monthly billing aren't seeing discounts, but annual billing users are" cuts the search space in half before Claude Code reads a single file.

**"I accepted the explanation without reading the fix."**
An explanation can be correct and a fix can still be wrong. Always read the diff — not just the explanation of the diff. Claude Code can explain a fix accurately while the implementation has an off-by-one, a missing null check at a different callsite, or an assumption that doesn't hold for edge cases your tests don't cover.

## 8. Trade-offs, Scale, and Edge Cases

On large codebases, full exploration is impractical in a single session. Claude Code will read what's reachable from the entry points you give it — but a 200-file module graph will involve many turns and risk missing non-obvious connections. The solution isn't to ask for more; it's to invest in a `CLAUDE.md` that encodes the architecture once, so Claude Code starts every session with a map already in hand.

For bugs in Azure-specific infrastructure (Function bindings, Service Bus triggers, Event Grid subscriptions), the execution model is different from a normal function call. The entry point is a binding, not a function call from another module. Always orient Claude Code explicitly: *"this is an Azure Function triggered by a Service Bus message — the entry point is the binding in function.json, not an import."*

For intermittent bugs and race conditions, Claude Code can reason about the code but cannot observe runtime timing. It can identify the *potential* race — where shared state is accessed without synchronisation — but confirming that this is the actual race in production requires runtime evidence (traces, logs with timestamps). Claude Code's role here is narrowing hypotheses, not closing the case.

## 9. Self-Check Questions

1. You open Claude Code on a codebase you've never seen and type: *"explain this codebase."* What's wrong with this approach, and how would you rewrite the request?

2. A colleague pastes a stack trace, Claude Code suggests adding a null check, your colleague ships it. What question should they have asked before accepting the fix?

3. You have a functional bug: the API returns the wrong total when a coupon and a loyalty discount are applied together. No errors. Unit tests pass for both discount types individually. How do you structure the Claude Code investigation to find the root cause?

4. A bug only appears in the Azure production environment but not locally or in staging. What information do you include in your Claude Code prompt that you wouldn't include for a local bug?

5. After Claude Code traces a bug and proposes a fix, it says: *"this change in OrderService.ts will resolve the issue."* What three things do you verify before approving the edit?

## Summary

Claude Code navigates and explains code the same way a senior engineer would: by following a thread, not reading everything. The quality of your exploration depends on the quality of your entry points — broad questions produce shallow maps, specific threads produce deep ones. For debugging, the most important habit is separating symptom from root cause: give Claude Code the trigger condition, not just the symptom, and push past the first plausible fix by asking it to explain the mechanism. Functional bugs live at module boundaries; technical bugs live at violated assumptions in the call chain. In both cases, the investigation is a conversation, not a single query.
