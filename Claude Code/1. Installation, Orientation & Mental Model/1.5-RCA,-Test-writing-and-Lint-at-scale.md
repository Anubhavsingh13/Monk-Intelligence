# Claude Code: Root Cause Analysis, Test Writing & Lint at Scale

**Last Updated:** 2026-04-15

> *Finding a bug is table stakes. Understanding why it exists, proving it can't come back, and cleaning the codebase it lived in — that's engineering.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Use Claude Code to perform root cause analysis that surfaces bug patterns, not just individual defects
- Identify the class of bug from a single instance and ask Claude Code to scan for related vulnerabilities
- Direct Claude Code to write tests for existing untested code — including edge cases you wouldn't have thought of
- Design an incremental regression test strategy that grows the test pack with every release, without a big-bang rewrite
- Fix an entire codebase's lint errors and formatting violations in a single focused session with minimal manual input
- Write a `CLAUDE.md` snippet that makes lint and test hygiene automatic for every future session

---

## 1. Why This Matters (In Our Systems)

Three scenarios. Each one is more common than it should be.

**Scenario A:** A production bug is fixed. The fix is correct. Two weeks later, the same root cause triggers in a different module because no one asked *why the bug was possible in the first place* — only *where it appeared this time*.

**Scenario B:** A critical service has 12% code coverage. The team knows it. Every sprint, someone adds a ticket to "add tests" that gets deprioritised. The coverage number doesn't move. Then a refactor breaks three behaviours silently and makes it to staging.

**Scenario C:** A legacy codebase has 847 lint warnings. No one fixes them because fixing them manually is soul-crushing work no one wants to own. The warnings become noise, and the three warnings that actually indicate a real bug are invisible in the flood.

Claude Code closes all three scenarios — not because it's clever, but because it can do the tedious work that humans keep deferring. The craft here is knowing how to direct it.

## 2. Intuition & Mental Models

### Root Cause Analysis: The Iceberg Model

A bug is the visible tip. Below the waterline:
- The **immediate cause**: the line that threw the error
- The **contributing cause**: the condition that made the error reachable
- The **systemic cause**: the pattern or assumption in the codebase that made this class of error possible

Most engineers fix the immediate cause and call it done. RCA means going two levels deeper. Claude Code can do this, but only if you ask for it explicitly. By default, it will fix the immediate cause — because that's what the error asked for.

### Test Writing: Coverage as a Ratchet

Think of your test suite as a ratchet. Every test you add is a click that can't go backward — a behaviour that's now locked in and will alert you if it ever regresses. The goal isn't 100% coverage in one sprint. The goal is making the ratchet click once per feature, once per bug fix, and once per release until the suite becomes genuinely protective.

Claude Code can write tests faster than any human. The skill is directing it to write tests that *actually protect* — not tests that achieve coverage numbers by asserting that a function returns without throwing.

### Lint Fixing: Batch, Don't Triage

Individual lint fix sessions are low-leverage. You fix three warnings, push, the CI pipeline stays yellow because 844 remain. The mental model that works: treat lint fixing as a single batch operation, not a continuous drip. One focused session, Claude Code in Auto mode with a minimal instruction, `git diff` at the end, done. The codebase goes from yellow to green in a morning.

## 3. Core Concepts & Terminology

**Root cause analysis (RCA)** — The process of tracing a defect back to the systemic condition that made it possible, not just the immediate code location where it manifested. A complete RCA produces a finding like: *"This class of null pointer is possible because we never validate optional fields at service boundaries — we assume the upstream always sends them."*

**Bug pattern** — A recurring class of defect produced by the same systemic cause. One null check missing is a bug. Fifteen null checks missing in the same pattern is a bug pattern — and the fix is a linting rule or a validation layer, not fifteen individual fixes.

**Regression test** — A test that encodes a specific behaviour or bug fix, preventing it from regressing in future changes. The best regression tests are written immediately after a bug is found, when the conditions are freshest.

**Incremental test strategy** — A policy of growing the test pack systematically rather than all at once. Common patterns: write one test per bug fixed, write tests for every new function added, require test coverage delta to be non-negative on every PR.

**Code coverage** — The percentage of code lines (or branches) executed by the test suite. Useful as a floor ("nothing should be less than X%"), harmful as a ceiling ("we hit X%, ship it"). Coverage tells you what's tested; it says nothing about whether the tests are meaningful.

**Lint** — Static analysis rules applied to source code to catch style violations, potential errors, and anti-patterns before runtime. Tools: ESLint (JavaScript/TypeScript), Pylint/Ruff (Python), Checkstyle (Java), Golangci-lint (Go). Lint errors are deterministic — the same code always produces the same warning — which makes them a perfect task for an automated agent.

**Auto-fixable lint errors** — A subset of lint rules that have a mechanical fix (e.g., missing semicolon, wrong quote style, unused import). These can be applied entirely automatically. Non-auto-fixable rules (e.g., "this function is too complex") require human judgement.

## 4. How It Works (What Actually Matters)

### RCA: What Claude Code Can and Can't Do

Claude Code can trace from a known bug to a systemic pattern by reading the codebase and identifying where the same assumption is made elsewhere. It does this by:

1. Reading the defective code and understanding the violated assumption
2. Searching the codebase for the same pattern (similar variable access, same missing check, same untrusted input used directly)
3. Summarising the pattern and how many locations share the same risk

What it can't do: observe runtime state, access production logs directly, or know which of those locations have been triggered by real traffic. RCA from Claude Code is a code-level audit — it tells you the full exposure, not which exposures have been exploited.

> ⚠️ **Counterintuitive:** The most valuable RCA output is not "here's the fix" — it's "here's the pattern and here are the other 11 places it exists." That list is what you take to a design review, not a PR.

### Test Writing: What Makes a Test Worth Having

Claude Code can generate tests quickly. The question is whether the tests it generates are protective. Three signals that a test is worth having:

1. **It would have caught the bug that just happened.** If you're writing tests after a bug, start here.
2. **It covers a branch the happy-path test doesn't reach.** Edge cases, error paths, boundary values.
3. **It encodes a business rule that isn't obvious from the code.** *"Discount must never be applied if the account is past-due"* — this should be a test, not a comment.

A test that asserts `expect(result).toBeDefined()` is coverage padding, not protection. Direct Claude Code away from this explicitly.

### Lint Fixing: Auto-Fix First, Triage Second

The highest-leverage lint session runs in two passes:

**Pass 1 — Auto-fix everything fixable:**

```bash
# For ESLint (TypeScript/JavaScript)
eslint --fix src/

# For Ruff (Python)
ruff check --fix src/
```

This handles 60–80% of warnings mechanically. Claude Code can run this in a bash command and report what's left.

**Pass 2 — Address remaining warnings with context:**

These are the non-auto-fixable warnings. Claude Code reads each one, understands the context, and either fixes the code or (if the warning is a false positive) adds a targeted suppression with an explanation comment.

> ⚠️ **Counterintuitive:** Suppressing a lint warning is not always wrong. A blanket `eslint-disable` file is wrong. A targeted `// eslint-disable-next-line @typescript-eslint/no-explicit-any -- external API type not yet typed` is engineering judgement. Claude Code can write these with appropriate comments — but only if you tell it to explain suppressions rather than silently adding them.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Root Cause Analysis — From One Bug to a Pattern

**The bug:** A null reference error when processing an order with no billing address. Fixed by adding a null check in `OrderProcessor.ts`.

**Stop here and ask for RCA:**

```
We just fixed a null reference on order.billingAddress in OrderProcessor.ts.
Before we close this, I want to understand the pattern.

1. What assumption was violated that made this possible?
2. Search the codebase for other places where order fields are accessed
   directly without a null/undefined check, especially in service and
   processor files.
3. Summarise the pattern: is this a one-off, or is there a systemic gap
   in how we handle optional order fields?

Do not fix anything yet — give me the audit first.
```

Claude Code reads `OrderProcessor.ts`, identifies the violated assumption (optional fields from the payment gateway are accessed as if required), searches for the same access pattern across `src/orders/` and `src/payments/`, and returns a list of 8 locations with the same exposure.

**Now you have a real finding.** You take it to the team. The outcome is a decision: add input validation at the service boundary, add a shared `validateOrderFields()` utility, or add a lint rule. One bug became a design conversation.

### Scenario 2: Writing Tests for Untested Code

**The situation:** `src/pricing/DiscountService.ts` has 0% test coverage. It's business-critical and about to be modified in the next sprint.

**Step 1 — Ask for a test strategy before writing tests:**

```
Read src/pricing/DiscountService.ts. Before writing any tests:

1. List every public method and what it does
2. For each method, identify: the happy path, the edge cases,
   and the error paths
3. Identify any business rules that are implicit in the code
   but not documented — things that would be non-obvious to a
   new engineer

Give me the analysis first. I'll approve it before you write any tests.
```

This step is critical. You're asking Claude Code to discover the test surface before writing tests. If it writes first, it will optimise for coverage. If it analyses first, you can steer it toward protection.

**Step 2 — Direct the test writing explicitly:**

```
Based on your analysis, write tests for DiscountService using Jest.

Rules:
- One describe block per method
- Test the happy path, every edge case you identified, and every
  error path
- For the "discount must not apply to past-due accounts" rule you found:
  write a test that would have failed before the rule was coded,
  to prove we're actually testing the business logic
- Do NOT write tests that only assert the return value is defined —
  every test must assert a specific value or a specific thrown error
- Place the test file at src/pricing/__tests__/DiscountService.test.ts
```

The explicit prohibition on `toBeDefined()` tests is the most important line. Without it, Claude Code will happily generate tests that hit coverage targets without protecting anything.

### Scenario 3: Incremental Regression Test Strategy

**The goal:** Grow the test pack with every release without a big-bang effort.

**Step 1 — Encode the strategy in `CLAUDE.md`:**

```markdown
## Testing Policy

When fixing a bug:
- Always write a regression test that would have caught the bug
  before the fix is merged. Name it: `it('should not [reproduce bug description]')`
- The test must fail on the unfixed code and pass on the fixed code

When adding a new function:
- Write at minimum: one happy path test, one null/undefined input test,
  one boundary value test

Coverage gate: PRs may not reduce overall coverage. New files must
have >= 80% coverage before merge.
```

**Step 2 — Ask Claude Code to enforce it on the current bug:**

```
We just fixed the billingAddress null bug. Per our testing policy
in CLAUDE.md, write a regression test for it in
src/orders/__tests__/OrderProcessor.test.ts.

The test must:
- Reproduce the exact condition that caused the bug (order with no billingAddress)
- Fail on the code before the fix
- Pass on the code after the fix
- Be named: "should handle order with missing billingAddress without throwing"
```

Over 10 sprints, this compounds. Each release adds 3–10 targeted tests. After six months, you have a test pack that maps directly to real production failures — the most valuable kind.

### Scenario 4: Fixing Lint at Scale

**The situation:** 847 ESLint warnings in a TypeScript codebase. Sprint planning keeps deferring it.

**One-session fix:**

```
This codebase has 847 ESLint warnings. Let's fix them all in this session.

Step 1: Run `npx eslint --fix src/` to apply all auto-fixable rules.
  Report how many warnings remain after.

Step 2: For remaining warnings, group them by rule name and count.
  Show me the top 5 rules by occurrence.

Step 3: For the top 5 rules, propose a fix strategy for each:
  - If it's a consistent style issue across the codebase, fix all instances
  - If it requires judgement (e.g., complexity warnings), show me 2 examples
    and ask how I want to handle the pattern
  - If it's a false positive for our use case, add a targeted suppression
    with a comment explaining why

Do not fix anything in step 2 or 3 until I approve the strategy.
```

The session structure matters:
- Step 1 is safe to auto-run (mechanical fixes only)
- Step 2 surfaces the true scope of what remains
- Step 3 asks for human judgement only where it's genuinely needed

The result: a morning's work, a green CI pipeline, and no manual triage.

## 6. Practical Usage & Decision Guidance

### RCA: When to Go Deeper

| Signal | Action |
|---|---|
| Bug in a module with no tests | RCA + write regression tests together |
| Same error type seen before | Ask Claude Code: "is this the same root cause as the last time?" |
| Bug at a module boundary | Ask for pattern audit across all similar boundaries |
| Bug introduced by a recent change | RCA + ask what assumption the change invalidated |
| Security-relevant bug (auth, data exposure) | Always RCA — never fix in isolation |

### Test Writing: What to Ask For, What to Avoid

| Ask for | Avoid |
|---|---|
| Tests that assert specific return values | Tests that only assert `toBeDefined()` or `not.toThrow()` |
| Tests named after the behaviour they protect | Tests named `test1`, `happy path`, `edge case` |
| One describe block per method | One massive describe block for the whole module |
| Tests for the exact condition that caused a bug | Tests that only cover the fixed code path |
| Explicit error path tests | Only testing success paths |

### Lint Fixing: Rule of Thumb

```
Auto-fixable warning  →  run --fix, no review needed
Consistent style gap  →  fix all instances, spot-check 3
Judgement-required    →  Claude Code proposes, human approves
Legitimate false positive  →  targeted suppression with comment
Genuine complexity warning  →  fix the code, not the warning
```

## 7. Common Pitfalls & Misconceptions

**"We fixed the bug, RCA is optional."**
The most expensive bugs are the ones that were fixed once before. If RCA is skipped, the same root cause will surface again — in a different location, at a worse time. The cost of a 20-minute RCA session is trivial compared to the cost of a second production incident.

**"High coverage means good tests."**
Coverage is a floor metric, not a quality signal. Claude Code can generate 90% coverage with tests that assert nothing meaningful. The question to ask before merging tests is: *"if someone deleted the logic these tests cover, would they fail?"* If the answer is "it depends," the tests are padding.

**"I'll add the regression test after the PR is merged."**
It never happens. The mental overhead of context-switching back to a closed bug is too high. The regression test belongs in the same PR as the fix. Claude Code makes this trivially fast — there's no excuse to defer it.

**"Running eslint --fix will break things."**
Auto-fixable rules are, by definition, safe mechanical transforms — semicolon insertion, quote normalisation, unused import removal. They cannot introduce logic changes. The only risk is a formatter that changes whitespace in a file a teammate has open. Run `git diff` after, review for surprises, commit. That's all the process you need.

**"I should suppress lint warnings I don't understand."**
Most people assume suppressing is a quick way to silence noise. The reality: a suppression without a comment is a debt note with no due date. Future-you and future-teammates have no idea if it was a genuine false positive or a lazy escape hatch. Claude Code will add comments to suppressions if you ask — the instruction is one sentence.

## 8. Trade-offs, Scale, and Edge Cases

RCA via code audit is excellent at finding structural patterns but blind to runtime behaviour. If a bug only manifests under specific load conditions, race conditions, or with specific production data distributions, static analysis won't surface it. Combine Claude Code's structural RCA with production telemetry (Application Insights, distributed traces) for the full picture.

For very large test suites (10,000+ tests), asking Claude Code to "add tests for this module" may produce tests that duplicate existing ones. Add a step: *"first check what tests already exist in `__tests__/` for this module, then write only what's missing."* This avoids redundancy and keeps the suite readable.

On lint at scale: if the codebase has never had consistent style enforcement, the first full lint session may produce a PR too large to review meaningfully. In this case, split by directory or by rule category — address formatting rules in one PR, potential-error rules in another. Claude Code can scope a fix session to a single ESLint rule across the entire codebase with one instruction.

## 9. Self-Check Questions

1. A colleague fixes a recurring null pointer bug for the third time in six months. Each time it's in a different file. What question should have been asked after the first fix, and how would you ask it with Claude Code?

2. You ask Claude Code to write tests for a module with 0% coverage. It produces 15 tests and coverage jumps to 87%. How do you evaluate whether those 15 tests are actually protective?

3. You want every future bug fix to come with a regression test. Where do you encode that policy so Claude Code applies it automatically, and what should the policy say?

4. Your codebase has 600 ESLint warnings. You run `eslint --fix` and 420 are resolved automatically. Of the remaining 180, how do you decide which ones Claude Code can fix autonomously vs. which ones need human review?

5. The RCA for a bug reveals 11 additional locations with the same vulnerability. How do you prioritise which to fix first, and what artifact do you produce from the audit to make sure none are forgotten?

## Summary

Root cause analysis, test writing, and lint fixing are the three most commonly deferred maintenance tasks in a codebase — because they're tedious, unglamorous, and easy to deprioritise. Claude Code eliminates the tedium in all three: it can trace a bug to its systemic cause in minutes, generate a protective test suite for an untested module in one session, and clear a backlog of hundreds of lint warnings with a handful of directed instructions. The craft is in knowing how to ask: RCA requires pushing past the immediate fix to the pattern; test writing requires explicitly prohibiting coverage padding; lint fixing works best as a two-pass batch operation rather than continuous triage. Encode the right policies in `CLAUDE.md` once, and these habits become the default — not the exception.
