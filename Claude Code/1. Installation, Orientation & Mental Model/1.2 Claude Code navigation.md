# Claude Code: Navigating Codebases, Context & Permissions
**Last Updated:** 2026-04-15

> *Knowing how to talk to Claude Code is the difference between a confused intern and a staff engineer who happens to type at 10,000 words per minute.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Start a Claude Code session and navigate an unfamiliar codebase using natural language questions
- Explain how Claude Code discovers and reads context — and steer it when it's looking in the wrong place
- Write prompts that guide Claude Code to the right files, modules, and entry points
- Understand the three permission tiers and choose the right one for your project context
- Configure `settings.json` to set default permission behaviour for your team
- Recognize when Claude Code has gone off-track and course-correct without starting over

---

## 1. Why This Matters (In Our Systems)

Picture this: you've just been handed a bug report in a service you've never touched. The stack trace points somewhere inside a 40-file module. Your options are to spend 45 minutes reading through the directory tree, or to open Claude Code and ask *"where does the payment retry logic live and how does it connect to the queue?"*

Thirty seconds later you're in the right file.

That's the promise. But here's the gap most engineers fall into: they open Claude Code, type a vague goal, and get a vague result. They conclude the tool isn't that useful. What they missed is that Claude Code reads context *on demand* — it doesn't have a map of your codebase pre-loaded. How you direct it determines how well it finds its way.

The permission model is the other gap. Engineers either leave it wide open (risky on shared systems) or so locked down that Claude Code asks for approval on every single file read (frustrating). Neither extreme is right. Understanding the model takes 10 minutes and saves a week of friction.

---

## 2. Intuition & Mental Models

Think of Claude Code as a very capable new engineer on their first day. They're smart, they can read any file you show them, and they're good at connecting dots. But they don't have the office map memorized. They need you to point them in the right direction, and they'll ask before doing anything that feels irreversible.

**Navigation:** If you tell them *"fix the login bug,"* they'll wander. If you tell them *"the login flow starts in `src/auth/login.ts` and the bug is somewhere in the token refresh logic,"* they'll be useful in two minutes. Same capability, wildly different output.

**Context:** Claude Code reads files the same way you'd read them during debugging — starting at an entry point and following the trail: imports, function calls, config references. It doesn't read everything at once. It reads what the problem requires. This is efficient, but it means it can miss things that aren't obviously connected to where it started.

**Permissions:** Imagine the new engineer has a personal policy: they'll read anything you point them to, but before they edit a file, delete something, or run a terminal command, they'll confirm with you first. That's the default permission model. You can loosen it (auto-approve edits) or tighten it (read-only, no terminal). The right setting depends on how much you trust the task and the environment.

## 3. Core Concepts & Terminology

**Session** — A single interactive run of Claude Code, starting when you type `claude` and ending when you exit. Context from previous sessions is not remembered unless captured in `CLAUDE.md`.

**Context discovery** — How Claude Code learns about your codebase. It reads files on demand as it reasons about a task — not upfront in bulk. It follows imports, searches for function definitions, reads config files, and uses directory listings to orient itself.

**Entry point** — The file or concept you give Claude Code to start from. A good entry point is like giving someone the right door to enter a building. A vague or wrong entry point means they spend time finding the right floor first.

**`CLAUDE.md`** — A Markdown file at the root of your project (or in subdirectories). Claude Code reads it automatically at the start of every session. It's your persistent briefing document: architecture notes, conventions, off-limits patterns, where things live.

**Permission model** — The set of rules that control what Claude Code can do without asking you first. Actions are grouped into: file reads, file writes, and shell command execution.

**`settings.json`** — Claude Code's configuration file, located at `~/.claude/settings.json` (global) or `.claude/settings.json` (project-level). This is where you set default permission behaviour.

**`/allowed-tools`** — A CLI flag and config key that explicitly lists which tools (read, write, bash) Claude Code is allowed to use in a session.

## 4. How It Works (What Actually Matters)

### Context Discovery in Practice

When you give Claude Code a task, it starts with what you've told it and expands outward:

1. It reads the files or paths you mentioned explicitly
2. It follows imports and references from those files
3. It searches for symbols, function names, or patterns you described
4. It reads `CLAUDE.md` if present
5. It uses directory listings to orient itself when it needs to explore

This is deliberate and efficient — but it means the quality of your starting direction matters a lot. Pointing it at the right module is faster than letting it search from scratch.

> ⚠️ **Counterintuitive:** Claude Code does not index your entire codebase on startup. There is no background scan. If you have a monorepo with 200 packages and you don't tell it which package is relevant, it will start guessing from the working directory. The more specific your entry point, the better the result.

### The Permission Model — Three Tiers

Every action Claude Code might take falls into one of three categories, each with a default behaviour:

| Action Type | Examples | Default |
|---|---|---|
| Read | Read a file, list a directory, search content | Auto-approved |
| Write | Edit a file, create a file, delete a file | Asks for confirmation |
| Execute | Run a shell command, install a package, run tests | Asks for confirmation |

The default is designed to be safe and visible: Claude Code will read freely but pause before it changes anything. You see a diff, approve it, and it proceeds.

You can adjust this per-session with flags, or permanently in `settings.json`.

> ⚠️ **Counterintuitive:** "Auto-approve writes" doesn't mean Claude Code will stop showing you diffs. It means it won't *wait* for your explicit confirmation before applying them. You can still interrupt mid-task. The distinction matters when running longer automated tasks.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Navigating an Unfamiliar Codebase

You've inherited a Python Flask API with no documentation. First session:

```
I've never worked in this codebase before. Start from app.py and give me
a tour: what are the main routes, where does business logic live, and what
external services does this app depend on?
```

Why this works: you gave it an unambiguous entry point (`app.py`), a concrete task (tour the structure), and specific things to look for (routes, logic, dependencies). Claude Code will read `app.py`, follow its imports, scan the route definitions, and build you a map.

Follow-up — drilling into a specific area:

```
The /payments route looks complex. Walk me through what happens when
a payment fails — specifically the retry logic. Start from the route handler.
```

You're guiding it like a pairing session, not issuing a single command and waiting.

### Scenario 2: Directing Context Explicitly

You have a monorepo with `packages/api`, `packages/worker`, and `packages/shared`. You're working on a bug in the worker:

```
Focus only on packages/worker for this session. The bug is in the job
scheduler — jobs are being enqueued twice. Start from packages/worker/src/scheduler.ts.
```

The phrase *"focus only on"* is doing real work here. It tells Claude Code not to wander into the API or shared packages unless explicitly asked. This saves time and prevents it from picking up patterns from unrelated code.

### Scenario 3: Setting Permissions for a Project

For a personal project where you trust the environment:

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(*)",
      "Write(*)",
      "Read(*)"
    ]
  }
}
```

For a shared staging environment where you want read-only exploration:

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Read(*)"
    ],
    "deny": [
      "Bash(*)",
      "Write(*)"
    ]
  }
}
```

For a production-adjacent codebase where you want writes but no shell execution:

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Write(*)"
    ],
    "deny": [
      "Bash(*)"
    ]
  }
}
```

The `*` is a wildcard matching all paths/commands within that action type. You can scope it further — for example `"Write(src/**)"` — to limit writes to a specific directory.

## 6. Practical Usage & Decision Guidance

**For codebase exploration:**

Start with broad orientation questions before narrow implementation tasks. A session that begins with *"help me understand this module"* produces better results than one that begins with *"fix the bug"* — because Claude Code has the context it needs before it acts.

**For directing context:**

| Situation | What to do |
|---|---|
| Unfamiliar codebase | Give an explicit entry point file |
| Monorepo with many packages | Scope the session to one package explicitly |
| Task spans multiple modules | Name each module upfront |
| Claude Code is reading wrong files | Say "stop reading X, focus on Y" mid-session |
| Recurring project context | Write it in `CLAUDE.md` once |

**For permissions:**

| Context | Recommended setting |
|---|---|
| Personal dev machine, trusted project | Allow all (Read, Write, Bash) |
| Team shared environment | Allow Read + Write, deny Bash |
| Reviewing or exploring only | Allow Read only |
| CI pipeline task | Allow only the specific tools needed |
| Production-connected environment | Deny Write and Bash by default |

## 7. Common Pitfalls & Misconceptions

**"I told it the bug and it went to completely the wrong place."**
Most people assume Claude Code knows the shape of their codebase. The reality: it starts from wherever you point it. If you say "fix the login bug" with no entry point, it will search for something called "login" — which might lead it to a test fixture, a frontend component, or a README before the actual auth module. Give it the file path.

**"It keeps asking permission for every single read."**
This usually means you've accidentally set `deny: Read(*)` or you're running in a very restrictive environment. File reads should be auto-approved by default. Check your `settings.json` — global settings in `~/.claude/settings.json` can override project-level settings in unexpected ways.

**"I gave it auto-approve and now I don't know what it changed."**
Auto-approving writes doesn't remove your responsibility to review. Run `git diff` after any agentic session. This is not a flaw in the tool — it's the same discipline you'd apply to any automated PR. Trust but verify.

**"It forgot what I told it in the last session."**
Claude Code has no memory across sessions by default. Everything you explained, every convention you mentioned, every "don't touch X" — gone when the session ends. This is what `CLAUDE.md` is for. If you've explained something more than once, it belongs in that file.

**"The `CLAUDE.md` isn't being read."**
Make sure it's in the directory where you launch Claude Code (the working directory), or in a parent directory. Claude Code walks up the directory tree looking for `CLAUDE.md` files and merges them, but it won't find one in a sibling or unrelated path.

## 8. Trade-offs, Scale, and Edge Cases

The on-demand context model is excellent for targeted tasks and terrible for tasks that require a global view of a large codebase. If you need Claude Code to understand the entire dependency graph of a 500-file monorepo, it will either miss things or spend a lot of turns reading before it acts. In these cases, a well-written `CLAUDE.md` that summarizes architecture and module ownership beats letting it discover everything organically.

The permission model trades safety for speed. Tight permissions (read-only, confirm everything) give you full visibility but slow down longer tasks significantly. Wide-open permissions (auto-approve all) are fast but require you to be disciplined about reviewing changes. Neither is universally right — the correct setting is the one that matches the risk profile of your environment.

On Windows (native, not WSL), the Bash tool has some limitations since it shells out to `cmd.exe` or PowerShell depending on configuration. If you're seeing unexpected shell execution failures, check whether your shell path in settings is pointing at the right interpreter. WSL users generally have a smoother experience with shell-heavy tasks.

## 9. Self-Check Questions

1. You're in a monorepo and ask Claude Code to fix a bug in the `payments` service. It starts reading files in the `notifications` service instead. What went wrong and how do you fix it?

2. A colleague asks "how does Claude Code know what my project does?" — what's the accurate answer, and what can you do to improve that knowledge?

3. You want Claude Code to freely edit source files but never run shell commands on a shared dev server. How do you configure this, and where does the config file go?

4. You've been using Claude Code for two sessions on the same project, and each time you have to re-explain the architecture. What's the right fix, and what should go in it?

5. What's the difference between Claude Code not having permission to do something and Claude Code asking for confirmation before doing something?

## Summary

Claude Code navigates your codebase on demand — it reads files as it needs them, starting from wherever you point it, not from a pre-built index. This makes it fast and efficient, but it means your entry points and context guidance directly shape the quality of its output. The `CLAUDE.md` file is your bridge between sessions: write it once, brief Claude Code every time. The permission model gives you three levers — read, write, execute — and the right configuration is the one that matches your environment's risk profile, not the most convenient default.
