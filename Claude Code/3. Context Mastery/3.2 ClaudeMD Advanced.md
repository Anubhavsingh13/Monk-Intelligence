# CLAUDE.md: What to Cut, What to Scope, and How to Stay Sharp

**Last Updated:** 2026-04-15

> *A bloated CLAUDE.md is as useless as no CLAUDE.md — useful instructions drown in noise, Claude Code wastes context on things it already knows, and no one updates it because it's too long to read.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Audit an existing `CLAUDE.md` and identify content that should be removed — because it's redundant, inferrable, or diluting signal
- Apply the "would Claude Code know this without me?" test to every instruction before keeping it
- Structure a project `CLAUDE.md` that stays under 150 lines and stays current
- Create a child `CLAUDE.md` in a specific module subdirectory with targeted, module-only instructions
- Explain when child `CLAUDE.md` files are the right solution vs. adding more to the root file
- Design a two-level `CLAUDE.md` architecture for a monorepo or multi-module project

---

## 1. Why This Matters (In Our Systems)

Here's a failure mode that takes about three months to develop:

Week 1: you write a tight `CLAUDE.md` — 60 lines, all signal. Claude Code sessions are noticeably better.

Week 4: you add a section about a new testing library. Reasonable.

Week 8: you add some architectural notes after a design review. Also reasonable.

Week 12: someone adds a long explanation of the payment module's history, why certain decisions were made in 2022, and a description of what the API is supposed to do. This is documentation, not instruction.

Week 16: `CLAUDE.md` is 400 lines. No one reads it before editing it. It contains three contradictory instructions about error handling that accumulated as different engineers added their preferences. Claude Code is following none of them consistently because the contradiction makes the instruction ambiguous. Engineers have stopped updating it because it feels like a maintenance burden.

The `CLAUDE.md` that started as a sharp tool has become a drawer full of things no one wants to throw away.

The discipline of *what not to put in* is the practice that keeps `CLAUDE.md` useful over time.

## 2. Intuition & Mental Models

### The Inference Test

Before any line goes into `CLAUDE.md`, ask one question:

*"Would Claude Code know this if it read the codebase carefully?"*

If yes — don't write it. If no — write it.

This is the core filter. Everything else flows from it.

Claude Code reads your code. It sees your `package.json`, your TypeScript config, your test files, your import statements. It can infer that you use Jest, that your project is TypeScript, that your components use a particular pattern — if those things are visible in the code.

What it cannot infer: the *why* behind decisions. Constraints that exist outside the code (Azure resource names, team policies, compliance requirements). Patterns you want applied that aren't in any existing file yet. The things you've consciously chosen *not* to do and why.

Those are `CLAUDE.md` material. Everything else is noise.

### The Newspaper Test for Instructions

Think of `CLAUDE.md` as a newspaper — not a novel. A newspaper headline is as short as it can be while still being unambiguous. A novel has context, backstory, nuance.

`CLAUDE.md` instructions should read like headlines. *"Never use console.log — use src/utils/logger.ts."* Not: *"In this project, we've historically preferred to use a centralised logging approach because we found that console.log statements in production created difficulty in our Azure Application Insights integration, so we created a logger utility..."*

The instruction is the first sentence. Everything after it is documentation that belongs in a different file.

### Child CLAUDE.md Files: The Right Scope at the Right Level

Think of `CLAUDE.md` files as layered briefings:

- **Global** (`~/.claude/CLAUDE.md`): things true about you as an engineer, across all projects
- **Project root**: things true about this entire project
- **Module subdirectory**: things true about this specific module that would be noise everywhere else

The subdirectory file is read only when Claude Code is working in or near that directory. It's invisible to the rest of the project. This makes it safe to be highly specific without cluttering the root file with module-only details.

## 3. Core Concepts & Terminology

**Signal-to-noise ratio** — The proportion of `CLAUDE.md` instructions that actually change Claude Code's behaviour vs. instructions that restate what Claude Code could infer, or that are too vague to apply. High signal = consistently better output. Low signal = no measurable effect, but longer read time and more confusion.

**Inferrable content** — Information Claude Code can derive by reading existing code, config files, or tests. Including this in `CLAUDE.md` is redundant at best, misleading at worst (if the code evolves but the `CLAUDE.md` doesn't update).

**Standing noise** — Instructions in `CLAUDE.md` that are too generic to change behaviour ("write clean code", "be careful with security"), outdated (referring to patterns or files that no longer exist), or contradictory with other instructions. Standing noise is the primary cause of `CLAUDE.md` entropy.

**Context scoping** — Placing instructions in the `CLAUDE.md` file closest to the code they apply to. Instructions about the payments module belong in `src/payments/CLAUDE.md`, not in the root file — where they're irrelevant for 90% of tasks.

**Context merge** — How Claude Code combines multiple `CLAUDE.md` files: root file provides baseline context, subdirectory file adds or overrides for that scope. Later files win on conflicts.

**Audit pass** — A deliberate review of `CLAUDE.md` content to identify and remove redundant, inferrable, contradictory, or stale instructions. Should happen quarterly, or after any significant architectural change.

## 4. How It Works (What Actually Matters)

### What Claude Code Can Infer Without Being Told

These things are already visible in your codebase — don't restate them in `CLAUDE.md`:

| Category | Example of redundant instruction | Why it's redundant |
|---|---|---|
| Language | "This project uses TypeScript" | `tsconfig.json` + `.ts` files make this obvious |
| Test framework | "We use Jest for testing" | `jest.config.ts` + `*.test.ts` files make this obvious |
| Package manager | "Use npm, not yarn" | `package-lock.json` presence vs. `yarn.lock` |
| Framework | "This is an Express app" | `import express from 'express'` in entry files |
| Import style | "Use ES module imports" | `"type": "module"` in `package.json` + existing `import` statements |
| File naming | "Components use PascalCase" | Existing component files |

Claude Code reads these files and adapts. Restating them wastes lines and adds false confidence that `CLAUDE.md` is comprehensive.

> ⚠️ **Counterintuitive:** Removing content from `CLAUDE.md` often *improves* Claude Code's behaviour. Fewer instructions means less contradiction, less ambiguity, and more focus on the things that actually require explicit guidance. Less is harder to write than more — but it's worth it.

### What Child CLAUDE.md Files Do

When Claude Code reads or writes files in `src/payments/`, it checks for a `CLAUDE.md` in that directory. If one exists, it's merged with the root context. Instructions in the child file apply *only* for work in or near that module.

The merge behaviour:
- Root file instructions apply everywhere
- Child file instructions apply when working in that directory (supplement or override)
- If a child file contradicts a root file instruction, the child file wins for that scope

This is how you can say "never throw unhandled errors" in the root file, but say "in the payments module, always throw a `PaymentError` with a transaction ID, never return an error code" in the child file — without those instructions fighting each other everywhere.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Auditing a Bloated CLAUDE.md

Here's a real-world `CLAUDE.md` that has accumulated entropy. We'll audit it line by line.

```markdown
# CLAUDE.md

This project is a TypeScript Node.js application.           ← REMOVE: inferrable from tsconfig.json
We use npm as our package manager.                          ← REMOVE: inferrable from package-lock.json
The entry point is src/index.ts.                            ← REMOVE: inferrable from package.json "main"
We use Jest for testing.                                    ← REMOVE: inferrable from jest.config.ts

We believe in writing clean, maintainable code.             ← REMOVE: too vague to apply
Code should be well-documented and easy to understand.      ← REMOVE: vague, unmeasurable

We use the logger at src/utils/logger.ts.                   ← KEEP: specific, non-inferrable constraint
Never use console.log or console.error.                     ← KEEP: explicit forbidden pattern

The payments module was rewritten in Q3 2023 because        ← REMOVE: history, not instruction
we had issues with the previous Stripe integration.
It now uses our internal PaymentGateway abstraction.        ← KEEP THIS PART: restate as instruction

Always use the PaymentGateway interface in                  ← KEEP: specific, actionable
src/payments/gateway/PaymentGateway.ts for all
payment operations. Never call Stripe directly.

We have unit tests and integration tests.                   ← REMOVE: inferrable from test file naming
Unit tests are in __tests__ directories.                    ← REMOVE: inferrable from test file locations
Integration tests end in .integration.test.ts               ← KEEP: naming convention is non-obvious

All PRs should be reviewed by at least one person.          ← REMOVE: this is a process doc, not Claude instruction
Please write descriptive commit messages.                   ← REMOVE: Claude Code doesn't write commits autonomously

The billing service connects to Azure.                      ← TOO VAGUE — rewrite or remove
```

**After the audit — what remains:**

```markdown
# CLAUDE.md

## Logging
Never use console.log or console.error. Use src/utils/logger.ts.

## Payments
Always use the PaymentGateway interface at src/payments/gateway/PaymentGateway.ts.
Never call the Stripe SDK directly from service or function code.

## Testing
Integration tests end in .integration.test.ts and must be run with npm run test:integration.
Do not run integration tests as part of the standard npm test suite.
```

From 28 lines to 11. Every line that remains changes behaviour. Every line removed was noise.

### Scenario 2: Adding a Child CLAUDE.md for the Payments Module

The payments module has specific rules that don't apply elsewhere — and the root `CLAUDE.md` is getting cluttered with payment-specific context. The fix: move it to `src/payments/CLAUDE.md`.

**Before (in root `CLAUDE.md` — adds 15 lines of module-specific noise):**

```markdown
## Payments Module
- Always use PaymentGateway interface
- Never call Stripe SDK directly
- All amounts in pence (integer), never float
- Use PaymentError for all errors in this module
- Transaction IDs must be logged on every operation
- Idempotency keys required for all charge operations
- The PaymentStatus enum is in src/payments/types/PaymentStatus.ts
- Refunds go through RefundService, never directly through PaymentGateway
```

**After — move to `src/payments/CLAUDE.md`:**

```markdown
# src/payments — Module Context

## Invariants
- All monetary amounts in pence (integer). Never use float for money.
- Always use PaymentGateway interface at gateway/PaymentGateway.ts.
  Never call Stripe SDK directly from service code.
- All charge operations require an idempotency key. See BasePaymentService.ts
  for the helper.

## Error Handling
- All errors thrown in this module must be instances of PaymentError.
  Include transactionId in every PaymentError. Never return error codes —
  always throw.

## Types
- PaymentStatus enum: src/payments/types/PaymentStatus.ts
- Never define new payment status values outside this enum.

## Refunds
- Refunds must go through RefundService. Never call PaymentGateway.refund()
  directly from function handlers or other services.
```

**Root `CLAUDE.md` now simply says:**

```markdown
## Payments
See src/payments/CLAUDE.md for module-specific rules.
All payment operations must go through the PaymentGateway interface —
never call payment SDKs directly from outside the payments module.
```

The root file retains the one cross-cutting rule (no direct SDK calls from outside the module). Everything else — which only matters when you're inside the payments module — lives where it's relevant.

### Scenario 3: Running Your Own Audit

Use this prompt to audit your `CLAUDE.md` in a Claude Code session:

```
Read the CLAUDE.md in this project root. For each instruction, evaluate:

1. Inferrable — Could you determine this from reading the codebase
   without being told? (package.json, config files, existing code patterns)
   If yes, mark it as removable.

2. Too vague — Is the instruction too generic to produce a consistent,
   specific behaviour change? ("write clean code", "be careful")
   If yes, mark it as removable OR suggest a specific rewrite.

3. Outdated — Does the instruction reference a file, pattern, or tool
   that no longer exists or has changed?
   If yes, mark it as removable or flag for update.

4. Wrong scope — Is the instruction specific to one module and cluttering
   the root file? 
   If yes, suggest which subdirectory CLAUDE.md it should move to.

5. Contradictory — Does any instruction conflict with another?
   If yes, identify the conflict and suggest the resolution.

Output a table: instruction | category | action | suggested rewrite (if applicable)
```

Run this quarterly. The result is a `CLAUDE.md` that stays lean, stays current, and keeps working.

### Scenario 4: CLAUDE.md Architecture for a Monorepo

For a project with three services (`orders`, `payments`, `notifications`) and shared infrastructure:

```
repo-root/
├── CLAUDE.md                    ← cross-cutting: shared conventions, forbidden
│                                   patterns, global testing policy
├── src/
│   ├── orders/
│   │   └── CLAUDE.md            ← orders-specific: Cosmos DB partition strategy,
│   │                               order state machine rules, event publishing
│   ├── payments/
│   │   └── CLAUDE.md            ← payments-specific: amount handling, Stripe rules,
│   │                               idempotency requirements
│   └── notifications/
│       └── CLAUDE.md            ← notifications-specific: provider interface rules,
│                                   retry policy, template naming
└── infra/
    └── CLAUDE.md                ← infra-specific: Bicep conventions, resource naming
                                    rules, what never to auto-modify
```

**Root `CLAUDE.md` — stays under 80 lines:**
- Logger path and forbidden `console.*`
- Layering invariant (functions → services → repositories)
- Secrets policy (Key Vault only)
- Testing: framework, file locations, coverage policy
- Azure environment summary (one paragraph)

**Module files — each under 50 lines:**
- Only what's unique to that module
- Cross-references to root for shared rules ("see root CLAUDE.md for logging policy")
- The one or two invariants that *must not be violated* within that module

The result: Claude Code working in `src/payments/` gets the full relevant context from both files. Claude Code working in `src/orders/` gets a completely different set of module-specific rules without seeing the payment noise — and vice versa.

## 6. Practical Usage & Decision Guidance

### The Audit Decision Tree

For each line in your `CLAUDE.md`:

```
Is it inferrable from reading the codebase?
  YES → Delete it.
  NO  → Continue.

Is it specific enough to produce a yes/no behaviour change?
  NO  → Rewrite it or delete it.
  YES → Continue.

Is it still accurate? (file exists, pattern still applies?)
  NO  → Update it or delete it.
  YES → Continue.

Is it relevant to the entire project, or just one module?
  ONE MODULE → Move to that module's CLAUDE.md.
  ENTIRE PROJECT → Keep it in the root.
```

### When to Create a Child CLAUDE.md

| Signal | Action |
|---|---|
| Root file has 3+ instructions specific to one module | Move them to a child file |
| A module has invariants that would contradict root instructions | Child file with local override |
| A module is maintained by a different team | Child file with team-specific context |
| An infra directory has different rules from application code | `infra/CLAUDE.md` |
| A module is being onboarded by new engineers | Child file with orientation context |

### Length Guidelines

| File | Target length | Hard limit |
|---|---|---|
| Global `~/.claude/CLAUDE.md` | 20–40 lines | 80 lines |
| Project root `CLAUDE.md` | 50–100 lines | 150 lines |
| Module child `CLAUDE.md` | 20–50 lines | 80 lines |

These aren't bureaucratic limits — they're the point at which `CLAUDE.md` stops being read and maintained. If you're near a limit, it's a signal to audit, not to expand the limit.

## 7. Common Pitfalls & Misconceptions

**"More instructions = better Claude Code behaviour."**
The reality: beyond a certain density, adding instructions reduces consistency. Claude Code must reconcile all instructions on every task. A vague instruction creates ambiguity that undermines a specific one nearby. Quality outperforms quantity every time.

**"I should document why we made decisions in CLAUDE.md."**
Decision history belongs in ADRs (Architecture Decision Records), PR descriptions, or code comments — not `CLAUDE.md`. Claude Code doesn't need the backstory. It needs the current rule. *"Always use PaymentGateway"* is the instruction. *"We rewrote this in Q3 after the Stripe incident..."* is documentation. They belong in different places.

**"Child CLAUDE.md files are for large projects only."**
Even a single-service project benefits from a child file when one module has meaningfully different rules. Size is not the signal. *Divergence from the root context* is the signal. If you find yourself prefacing instructions with "but only in the payments module...", that's a child file.

**"I don't need to audit CLAUDE.md — I'll notice when it drifts."**
You won't. `CLAUDE.md` entropy is gradual and invisible. Each addition seems reasonable. The contradictions accumulate slowly. Engineers stop noticing because each correction seems like a one-off. The quarterly audit is the mechanism that catches what daily usage misses.

**"Moving instructions to a child file means the root file won't apply there."**
Root file instructions still apply everywhere, including in subdirectories. The child file *adds* or *overrides* — it doesn't replace. If you moved everything from the root to child files, the root instructions would still apply unless explicitly overridden. This is the merge behaviour — addition, not substitution.

## 8. Trade-offs, Scale, and Edge Cases

The child `CLAUDE.md` approach assumes Claude Code is launched from the repo root. If an engineer always `cd`s into a module before running `claude`, the root file may not be read (depending on how far up the tree Claude Code walks). Establish a team norm: always launch from the project root, let `CLAUDE.md` hierarchy do its job.

For very large files that legitimately need many rules: use clear section headers and keep sections short. A 150-line `CLAUDE.md` with eight 15-line sections is readable. A 150-line `CLAUDE.md` as a wall of prose is not. Structure aids both human readability and Claude Code's ability to prioritise relevant instructions for a given task.

When two engineers have conflicting conventions and both are encoded in `CLAUDE.md`, Claude Code will apply them inconsistently. This is a symptom of an unresolved team disagreement that was deferred rather than decided. `CLAUDE.md` can't resolve a conflict that the team hasn't — it can only surface it. The fix is a team decision, not a `CLAUDE.md` edit.

On version control: `CLAUDE.md` changes should go through PR review like any other project file. A `CLAUDE.md` change that encodes a new convention without team buy-in is a governance problem in file form.

## 9. Self-Check Questions

1. Your `CLAUDE.md` contains: *"This project uses React and TypeScript for the frontend."* Apply the inference test. What's your verdict, and what would you do with this line?

2. You have 12 lines in the root `CLAUDE.md` about how the billing module handles failed invoices, retries, and dunning state transitions. No other module has these concepts. What's the right action, and where does that content go?

3. A colleague argues that removing inferrable content from `CLAUDE.md` is risky because "what if Claude Code misreads the codebase?" How do you respond?

4. You run the audit prompt and discover two instructions in the root `CLAUDE.md` that directly contradict each other on error handling. Both were added by different engineers six months apart. What's the process to resolve this, and what's the risk if you don't?

5. Your monorepo has a `src/auth/` module with very strict security requirements — no external calls, no logging of token values, specific error codes for specific failure cases. How do you structure the `CLAUDE.md` so these rules apply precisely when Claude Code touches auth code, without cluttering sessions that don't touch auth at all?

## 10. What to Learn Next

- **Hooks & Automation** — `CLAUDE.md` is policy stated as prose. Hooks enforce policy as code — running linters, blocking writes, executing tests automatically. The two are complementary: `CLAUDE.md` handles the judgement calls, hooks handle the mechanical enforcement.
- **Output Quality Auditing** — The quarterly `CLAUDE.md` audit and the weekly output audit are the same discipline applied at different timescales. Running both gives you a compounding feedback loop: session corrections feed the weekly audit, weekly audit findings feed the quarterly `CLAUDE.md` cleanup.
- **Multi-Agent Workflows** — In multi-agent setups, child `CLAUDE.md` files become especially important: each sub-agent working in a specific module gets precisely scoped context. Understanding the hierarchy is the prerequisite for designing multi-agent context architecture.

## Summary

A `CLAUDE.md` that contains only what Claude Code cannot infer is more powerful than one that contains everything you know about the project. The inference test — *"would Claude Code know this from reading the codebase?"* — is the primary filter that keeps the file lean and effective. Child `CLAUDE.md` files in module subdirectories apply module-specific rules precisely where they're relevant, without polluting the root context for unrelated tasks. Both disciplines require the same habit: deliberate curation over accumulation. Write less. Update regularly. The file that gets maintained is the file that stays useful.
