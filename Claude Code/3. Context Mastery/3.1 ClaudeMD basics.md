# CLAUDE.md: Persistent Context & Project Memory

**Last Updated:** 2026-04-15

> *Every Claude Code session starts cold — unless you wrote a CLAUDE.md. That one file is the difference between briefing a new contractor every morning and working with a colleague who already knows the codebase.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Explain precisely what `CLAUDE.md` is, what it does to a Claude Code session, and what it cannot do
- Run `/init` on a real project and evaluate the output Claude Code auto-generates
- Identify the four categories of content that belong in `CLAUDE.md` — and the content that doesn't
- Write a complete `CLAUDE.md` by hand for a project you own, structured so instructions are clear, specific, and non-contradictory
- Maintain `CLAUDE.md` as a living document — updating it from session corrections, audit findings, and architecture changes
- Understand the hierarchy when multiple `CLAUDE.md` files exist (global, project root, subdirectory)

---

## 1. Why This Matters (In Our Systems)

Here's something that happens to every engineer in their first month with Claude Code:

Session 1: you tell Claude Code *"we use the logger at `src/utils/logger.ts`, never `console.log`."* It complies. Good session.

Session 2: new task, new session. Claude Code uses `console.log` again. You correct it again. It complies again.

Session 5: same correction. Same compliance. Same forgetting next time.

The problem isn't the model. The problem is that Claude Code has no memory across sessions. Everything you explained yesterday is gone today. If you have five engineers on a team and each corrects the same thing twice a week, that's fifty context-restoration minutes a week that produce zero durable value.

`CLAUDE.md` is the fix. It's a file Claude Code reads automatically at the start of every session. Conventions, architecture decisions, forbidden patterns, environment context — written once, applied forever. It converts your hardest-won project knowledge into standing instructions that don't need to be re-explained.

The engineers who get the most from Claude Code on a real codebase are the ones who invested an hour writing a good `CLAUDE.md` in week one. It compounds from the first session it applies.

## 2. Intuition & Mental Models

### The Onboarding Document Analogy

Think about what you give a new engineer on day one: codebase tour, architecture overview, conventions to follow, tools to use, patterns to avoid, deployment process. You spend 90 minutes walking them through it. Without that briefing, they'd spend a week making avoidable mistakes.

`CLAUDE.md` is that briefing — but for Claude Code. Every session is Claude Code's "day one." `CLAUDE.md` is what makes it not feel that way.

The critical difference from the new engineer analogy: a human engineer learns from corrections and retains that learning. Claude Code retains nothing across sessions unless you write it down. So `CLAUDE.md` isn't a one-time briefing — it's a living briefing that grows every time you encounter a correction worth making permanent.

### What CLAUDE.md Is and Isn't

**It is:**
- A plain Markdown file Claude Code reads at session start
- Persistent context that applies to every task in that project
- Your mechanism for encoding project-specific conventions, constraints, and knowledge
- A team-level asset — commit it to version control and every engineer gets the same Claude Code behaviour

**It isn't:**
- A replacement for good instructions in the session prompt
- A prompt that makes Claude Code "smarter" in general
- Documentation for humans (though it can double as that)
- A configuration file — it has no special syntax, just Markdown Claude Code reads and acts on

> The most common misconception: *"I'll put everything in `CLAUDE.md` and I won't need to write good prompts."* `CLAUDE.md` provides baseline context. It doesn't replace specificity in individual tasks. Think of it as background knowledge, not active instruction.

## 3. Core Concepts & Terminology

**`CLAUDE.md`** — A Markdown file that Claude Code reads automatically at the start of every session. It can live at the project root, in subdirectories, or globally at `~/.claude/CLAUDE.md`. The content is treated as contextual instructions that persist for the entire session.

**`/init`** — A Claude Code slash command that analyses your project and generates a starting `CLAUDE.md` automatically. It reads package files, directory structure, config files, and existing documentation to detect conventions, tools, and patterns. The output is a draft — always review and edit before using.

**Global `CLAUDE.md`** — Located at `~/.claude/CLAUDE.md`. Applied to every project on your machine. The right place for personal preferences (your preferred editor workflow, global conventions) that apply everywhere.

**Project `CLAUDE.md`** — Located at the repository root. Applied to all sessions launched in that directory. The right place for project-specific conventions, architecture, and constraints. Should be committed to version control.

**Subdirectory `CLAUDE.md`** — Located inside a subdirectory (e.g., `src/payments/CLAUDE.md`). Applied when Claude Code is working in or reading from that directory. The right place for module-specific context in large codebases.

**Context hierarchy** — When multiple `CLAUDE.md` files exist, Claude Code reads and merges them: global first, then project root, then subdirectory closest to the working context. Later files can extend or override earlier ones.

**Standing instruction** — A directive in `CLAUDE.md` that applies automatically to every session without requiring re-entry. The product of a correction worth making permanent.

## 4. How It Works (What Actually Matters)

### How Claude Code Reads CLAUDE.md

When you run `claude` in a directory, Claude Code:
1. Checks for `~/.claude/CLAUDE.md` (global) and reads it
2. Walks up the directory tree from your working directory, collecting every `CLAUDE.md` it finds
3. Reads them in order from root to current directory
4. Merges the content into its working context before your first message

This happens before you type anything. The instructions in `CLAUDE.md` are effectively pre-loaded into the session's context. From Claude Code's perspective, they read like a briefing you gave at the start of the conversation.

> ⚠️ **Counterintuitive:** Claude Code doesn't parse `CLAUDE.md` as structured config — it reads it as natural language. This means section headers, bullet points, and clear prose matter. Ambiguous or contradictory instructions in `CLAUDE.md` produce inconsistent behaviour, not errors. Write it as you'd write a clear policy document, not as a config file.

### What Claude Code Does With the Content

Every instruction in `CLAUDE.md` is evaluated against each task. Specific, actionable instructions ("never use `console.log`") apply cleanly and consistently. Vague instructions ("write clean code") produce inconsistent results because Claude Code has to interpret what they mean in each context.

The sections that have the highest impact are:
1. **Explicit forbidden patterns** — things Claude Code should never do
2. **Environment and toolchain facts** — runtime versions, SDK choices, deployment model
3. **File and module ownership** — what lives where, what each module is responsible for
4. **Testing policy** — what test runner, what coverage expectations, what test patterns

Instructions that describe *constraints* ("never X") are more reliable than instructions that describe *preferences* ("prefer X") because they're unambiguous. When the preference matters, state it as a constraint: *"always use X, never use Y."*

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Running /init and Evaluating the Output

Navigate to a real project you own:

```bash
cd path/to/your/project
claude
```

Inside the session:

```
/init
```

Claude Code will read your project structure — `package.json`, framework configs, directory layout, existing READMEs — and generate a draft `CLAUDE.md`. Here's what a typical output looks like for a Node.js Azure Functions project:

```markdown
# Project Context

## Tech Stack
- Runtime: Node.js 20, Azure Functions v4
- Language: TypeScript 5.x
- Test framework: Jest
- Package manager: npm

## Project Structure
- `src/functions/` — Azure Function trigger definitions
- `src/services/` — Business logic
- `src/utils/` — Shared utilities
- `infra/` — Bicep infrastructure templates

## Development Commands
- `npm run start` — Start local Azure Functions host
- `npm test` — Run Jest test suite
- `npm run lint` — Run ESLint
```

**Evaluate the output by asking:**

1. **Is the tech stack accurate?** `/init` reads `package.json` but may miss runtime version specifics or recently added dependencies. Verify.
2. **Is the structure correct?** Auto-detection is good at finding directories but bad at understanding what each module *owns* vs. what it *contains*. Add ownership descriptions.
3. **Are the commands right?** Claude Code may have found scripts in `package.json` but might have missed project-specific environment setup needed before commands work (e.g., `az login`, `func core tools version`).
4. **What's missing?** Conventions, forbidden patterns, architectural decisions, testing policy — these almost never appear in `/init` output because they're not in config files. They live in engineers' heads. This is what you add by hand.

Treat `/init` output as a scaffold, not a finished document. It saves you the boilerplate. You provide the knowledge.

### Scenario 2: Writing CLAUDE.md by Hand — A Complete Example

For a Node.js API deployed as Azure Functions with TypeScript:

```markdown
# CLAUDE.md — Order Processing Service

## Project Overview
Node.js TypeScript API deployed as Azure Functions v4 (HTTP + Service Bus triggers).
Connects to: Azure Cosmos DB (orders), Azure Service Bus (event publishing),
Azure Key Vault (secrets), Application Insights (telemetry).

---

## Development Environment

### Prerequisites
- Node.js 20.x
- Azure Functions Core Tools v4: `npm install -g azure-functions-core-tools@4`
- Azurite (local storage emulator): `npm install -g azurite`

### Commands
- `npm run dev` — Start local Functions host (requires Azurite running)
- `npm test` — Run Jest suite
- `npm run test:integration` — Run integration tests (requires real Azure connection)
- `npm run lint` — ESLint with auto-fix
- `npm run build` — TypeScript compile check

---

## Architecture

### Module Ownership
- `src/functions/` — Azure Function entry points ONLY. No business logic here.
- `src/services/` — All business logic. Services are the only callers of repositories.
- `src/repositories/` — All Cosmos DB access. No business logic here.
- `src/events/` — Service Bus publishing. Consumed only by services, never by functions directly.
- `src/middleware/` — Express-compatible middleware (auth, logging, correlation ID).
- `infra/` — Bicep templates. Never auto-modify without explicit instruction.

### Key Invariants
- Functions call services. Services call repositories and events. Repositories call Cosmos DB.
  This layering must not be violated — no function may call a repository directly.
- All secrets come from Key Vault via the `src/config/secrets.ts` loader. Never hardcode
  secrets or read from environment variables directly in business logic.

---

## Coding Conventions

### Logging
- NEVER use `console.log`, `console.error`, or `console.warn`.
- Use the logger at `src/utils/logger.ts` for all logging.
- Log structured objects, not string concatenation.
- Always include `correlationId` in log entries when available.

### Error Handling
- All errors thrown from services must be instances of `AppError` from `src/errors/AppError.ts`.
- HTTP error responses must use `src/middleware/errorHandler.ts` — never construct error
  response objects manually in function handlers.
- Never swallow errors silently. If you catch and don't rethrow, log the error first.

### TypeScript
- No `any` types. If a type is unknown, use `unknown` and narrow explicitly.
- All function parameters and return types must be explicitly typed.
- Prefer `interface` over `type` for object shapes that will be implemented or extended.

### Azure Functions
- Use Azure Functions v4 programming model syntax (app.http(), app.serviceBusTopic(), etc.)
- NOT the v3 function.json approach.
- All HTTP triggers must use `authLevel: 'function'` unless explicitly specified otherwise.

---

## Testing Policy

- Test runner: Jest with ts-jest
- Test files: co-located in `__tests__/` subdirectories, named `*.test.ts`
- Every bug fix must include a regression test that would have failed before the fix
- Every new service method must have: one happy path test, one error path test
- Do NOT write tests that only assert `.toBeDefined()` or `.not.toThrow()`
- Do NOT mock the Cosmos DB client in unit tests — use the in-memory mock in `src/test-utils/`
- Integration tests in `*.integration.test.ts` — these hit real Azure resources and must be
  run separately with `npm run test:integration`

---

## Forbidden Patterns

- No `console.*` — use the app logger
- No direct `process.env` access in business logic — use `src/config/` loaders
- No `require()` — this is an ESM project, use `import`
- No inline SQL or NoSQL queries in service files — all data access through repositories
- No new npm packages without noting the reason in the PR description
- Do not modify `infra/` Bicep files unless the task explicitly requires infrastructure changes

---

## Azure-Specific Context
- Region: West Europe (all resources in `rg-ordersvc-prod-westeu`)
- Cosmos DB: SQL API, partition key is `/tenantId` on all collections
- Service Bus: Standard tier, topics not queues. Dead-letter queue is enabled on all subscriptions.
- Key Vault references are resolved at Function startup — secrets are not re-fetched per-request.
- Application Insights: structured logging only. Use `src/telemetry/client.ts` for custom events.
```

This is a real working `CLAUDE.md`. Notice what each section does:
- **Project Overview** — eliminates the most common orient-from-scratch guesses
- **Architecture / Module Ownership** — the invariant Claude Code must never violate
- **Coding Conventions** — specific, stated as constraints, not preferences
- **Testing Policy** — the exact rules from the previous wiki page, persisted here
- **Forbidden Patterns** — the negative constraints that prevent the most common mistakes
- **Azure-Specific Context** — facts about the infrastructure that aren't in the code

### Scenario 3: Growing CLAUDE.md from Session Corrections

After every significant Claude Code session, ask this:

```
What corrections did I make in this session that are worth making 
permanent? For each, suggest a specific line or section I should 
add to CLAUDE.md.
```

Over one week, a real team collected these additions:

```markdown
# Added after week 1 corrections

## Conventions (additions)
- Cosmos DB query results must always be mapped through the repository's
  `toModel()` method before being returned to service callers.
  Never return raw Cosmos DB documents from repositories.

## Testing Policy (additions)
- When testing Service Bus event publishers, use the mock at
  `src/test-utils/mockServiceBus.ts`. Do not import the real SDK in unit tests.

## Azure-Specific Context (additions)
- The Cosmos DB client uses optimistic concurrency via `_etag`. When updating
  documents, always pass the etag to prevent lost-update races.
  See `src/repositories/BaseRepository.ts` for the pattern.
```

None of these were invented. Each was a correction made in a session that, without `CLAUDE.md`, would be made again next week.

## 6. Practical Usage & Decision Guidance

### What Belongs in CLAUDE.md

| Category | What to include | What to leave out |
|---|---|---|
| **Environment** | Runtime versions, required tools, setup steps | Instructions that change per-developer (personal API keys, local paths) |
| **Architecture** | Module ownership, layer invariants, key dependencies | Implementation details that belong in code comments |
| **Conventions** | Specific forbidden patterns, required patterns, naming rules | Generic advice ("write clean code", "be careful") |
| **Testing** | Test runner, file locations, coverage policy, specific forbidden test patterns | Test data or fixtures (those belong in test files) |
| **Azure/Infra** | Resource names, SDK versions, partition keys, connection patterns | Secrets, connection strings, credentials |
| **Workflow** | PR conventions, branch naming, commit message format | External process docs (link to them instead) |

### CLAUDE.md vs. Session Prompt

| Put it in CLAUDE.md when... | Put it in the session prompt when... |
|---|---|
| It applies to every task in this project | It's specific to this task |
| It would need to be re-explained next session | It's context for this session only |
| A teammate would also benefit from it | It's a one-time clarification |
| It came from a correction you've made more than once | You've only needed it once |

### The Three-Section Minimum

If you write nothing else, these three sections have the highest leverage:

1. **Forbidden Patterns** — the 5–10 things Claude Code should never do in this project
2. **Module Ownership** — what lives where and the layer invariants
3. **Testing Policy** — what framework, where test files go, what makes a test acceptable

A `CLAUDE.md` with just these three sections, written specifically, will eliminate 70% of the corrections most teams make.

## 7. Common Pitfalls & Misconceptions

**"I ran /init and the CLAUDE.md looks good — I'm done."**
`/init` detects tooling and structure. It cannot detect conventions, architectural decisions, forbidden patterns, or the context behind your module boundaries. These are the most valuable things in a `CLAUDE.md` and they're almost never in `/init` output. `/init` is the starting block, not the finish line.

**"I'll make CLAUDE.md comprehensive — everything Claude Code should know."**
Volume is not the goal. Specificity is. A 50-line `CLAUDE.md` with precise, actionable constraints outperforms a 300-line one that mixes clear rules with vague preferences and aspirational descriptions. If an instruction can't be applied to a specific file or task in a clear yes/no way, it probably doesn't belong.

**"The CLAUDE.md is written — it doesn't need updating."**
A `CLAUDE.md` that doesn't get updated drifts from reality. Architectural changes, new dependencies, new conventions — these all need to be reflected. The test: if a correction appears in two consecutive sessions, it belongs in `CLAUDE.md`. If it appears there already, the wording may need to be more specific.

**"Personal preferences should go in the project CLAUDE.md."**
Instructions that reflect your personal workflow (e.g., "always show me the full diff before applying") belong in `~/.claude/CLAUDE.md` (global), not the project root. Project `CLAUDE.md` is committed to version control — your teammates get it too. Only project-wide conventions belong there.

**"CLAUDE.md handles everything — I can write vague prompts now."**
`CLAUDE.md` provides background context. It doesn't replace task-specific precision. *"Add rate limiting"* is still a vague prompt even with a perfect `CLAUDE.md`. The brief still needs a clear goal, constraints, and scope. `CLAUDE.md` handles the *always-true* context; your prompt handles the *task-specific* context.

## 8. Trade-offs, Scale, and Edge Cases

In a monorepo with many services, a single root `CLAUDE.md` will be too generic to be useful for any specific service. The subdirectory `CLAUDE.md` pattern solves this: a root file covers shared conventions, each service directory has its own file covering service-specific context. Claude Code merges them when working in that directory.

For open-source projects or public repositories, `CLAUDE.md` will be visible to everyone who clones the repo. This is usually fine — it's just conventions. But don't put anything in it that reveals internal architecture you'd prefer to keep private, and never put connection strings, resource names that reveal your production topology, or security-relevant patterns.

The `/init` command's quality depends heavily on how well-structured your project is. For a fresh, well-organised project with clear naming conventions, `/init` produces a useful scaffold. For a legacy codebase with inconsistent structure, `/init` output may be misleading — it reflects what the files look like, not what they were intended to be.

For teams adopting Claude Code: the first `CLAUDE.md` should be a team exercise, not an individual one. The conventions it encodes represent team decisions. Writing it alone and committing it is technically correct but misses the opportunity to surface and resolve the conventions that different team members have been applying inconsistently.

## 9. Self-Check Questions

1. A teammate runs `/init`, reviews the output, and says: *"Looks good — covers everything."* What category of content is almost certainly missing, and how would you explain why `/init` can't detect it?

2. You write the instruction: *"Always write clean, readable code."* Why will this not reliably change Claude Code's behaviour, and how would you rewrite it as an effective `CLAUDE.md` instruction?

3. You correct Claude Code in a session for the third time for the same pattern — using raw Cosmos DB documents instead of mapping through `toModel()`. Where does the fix live, and what specifically do you write?

4. Your project has three services in a monorepo: `orders`, `payments`, `notifications`. Each has different conventions and dependencies. How do you structure your `CLAUDE.md` files, and what goes in each layer?

5. A new engineer joins the team and asks: *"What's the difference between CLAUDE.md and the README?"* What's your answer?

## Summary

`CLAUDE.md` is the file that makes Claude Code feel like a colleague rather than a contractor on day one — every single session. It's read automatically at session start, applies to every task, and costs nothing to maintain beyond the discipline of updating it when you make a correction worth keeping. `/init` generates a structural scaffold; the valuable content — forbidden patterns, module ownership, testing policy, architectural invariants — comes from you. Write `CLAUDE.md` as clear constraints, not vague preferences, commit it to version control so the whole team benefits, and update it every time you correct the same mistake twice. One hour of investment in the first week compounds across every session that follows.
