# Claude Code: Context Window Management & Prompt Precision

**Last Updated:** 2026-04-15

> *Context is the most expensive resource in a Claude Code session. The engineer who manages it well completes 10x more work per session than the one who wastes it pasting files.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Diagnose when a session has degraded due to context window saturation — and know how to recover
- Decide strategically when to use `/clear` to start a fresh session vs. continuing with existing context
- Write instructions that tell Claude Code to *fetch* the context it needs rather than you pasting files
- Rewrite vague prompts into specific, constrained ones — and quantify the output quality improvement
- Structure a multi-step task so that context is never wasted on irrelevant information
- Recognize the trade-off between conversation continuity and context efficiency

---

## 1. Why This Matters (In Our Systems)

Here's a real failure mode that costs hours:

A session starts with a clear goal. You make five task requests. Claude Code produces good output. On task six, the output is suddenly confused — recommending patterns from the first task that aren't relevant to the sixth. The response is slower. The reasoning is shallow.

What happened: the context window filled up. Claude Code is still operating, but it's running on a compressed, degraded view of the conversation. Earlier context is falling out of the window. The model can't "see" the current file structure anymore because it sacrificed that space to remember the first three tasks.

The engineer blames the model. The reality: the engineer overloaded the context window.

The fix isn't a bigger context window (though that helps). It's knowing *why* context fills up, *when* to start fresh, and *how* to write prompts that pull context efficiently instead of pushing it wastefully.

---

## 2. Intuition & Mental Models

### Context as a Budget, Not Infinite

Think of the context window as a budget. Every token you spend — task description, file contents you paste, conversation history — reduces what's available for Claude Code's reasoning about the *current* task.

The engineer who spends wisely: they instruct Claude Code to read the three files it needs, not pasting all 47 files in the module. They give a precise goal, not a vague one that requires Claude Code to guess what's needed. They start a fresh session when the budget for the current task is too small relative to remaining tasks.

The engineer who wastes: they paste entire config files when the prompt could have said "read `.env.example` to understand the environment variables." They write vague tasks that require Claude Code to spend context tokens figuring out what was actually asked. They let a session bloat with conversation history that stopped being relevant five tasks ago.

### "Pull vs. Push" Prompting

**Push prompting** — you paste files into the prompt, then ask Claude Code to act on them.

```
Here's my OrderService.ts:
[entire file pasted]

Here's my OrderRepository.ts:
[entire file pasted]

Now add a caching layer.
```

This wastes context. You've pre-loaded files Claude Code might not need. It uses space in the context window that could go to reasoning.

**Pull prompting** — you tell Claude Code what it needs and let it fetch.

```
Add a caching layer to the order processing flow. Start by reading
src/services/OrderService.ts and src/repositories/OrderRepository.ts
to understand the current flow. Then propose a caching strategy.
```

Claude Code reads exactly the files it needs, when it decides it needs them. The context window is used for reasoning, not for files you preemptively pasted.

The second approach is cheaper and produces better output — because more context is available for thinking.

### When to /clear: Continuity vs. Efficiency Trade-Off

A session accumulates context. Early on, continuity matters — Claude Code remembers your project, conventions, style. But as the session grows, old context becomes noise. The trade-off:

- **Keep the session** if the next task benefits from remembering previous tasks or from `CLAUDE.md` loaded at session start
- **Start fresh with `/clear`** if the next task is unrelated and the conversation history has accumulated 20+ exchanges

The signal: *does the current accumulated context help or hurt the next task?* If it helps, continue. If it hurts, `/clear`.

## 3. Core Concepts & Terminology

**Context window** — The amount of conversation history Claude Code can "see" at once. Measured in tokens. Limited but large; not unlimited. Careful management extends what you can accomplish in one session.

**Token** — The unit of information Claude Code processes. Roughly 4 characters or 1 word. A 2,000-line file is ~3,000–5,000 tokens.

**Context saturation** — The state where the conversation history has grown so large that Claude Code's reasoning capacity on the *current* task is noticeably degraded. Symptoms: shallow reasoning, repetition, ignoring recent instructions, recommending irrelevant patterns.

**Pull prompting** — Instructing Claude Code to fetch what it needs (read files, search, explore) rather than you pasting contents upfront.

**Push prompting** — You providing all relevant context upfront by pasting files into the prompt.

**Session continuity** — The value of keeping a session open across multiple related tasks so that Claude Code retains context from earlier work (project structure, conventions, earlier corrections).

**Context freshness** — The degree to which the currently loaded context is relevant to the current task. High freshness = old context can be discarded. Low freshness = old context is still useful.

**Prompt precision** — The specificity with which you describe a goal. "Add a feature" is low precision. "Add rate limiting to the POST /orders endpoint using the existing RateLimiter middleware, configured via App Configuration, with 100 req/min per tenant" is high precision.

## 4. How It Works (What Actually Matters)

### How Context Gets Wasted

**Scenario 1 — Pasting files unnecessarily:**
```
Here's my entire middleware stack [4,000 tokens]:
[whole file pasted]

Now I want to add CORS headers to the auth middleware.
```

Claude Code needed to see only the auth middleware section — maybe 200 tokens. The other 3,800 tokens of middleware it didn't need filled the budget.

**Scenario 2 — Vague tasks that require context-burning exploration:**
```
Improve the error handling in this codebase.
```

Claude Code must now burn context exploring the codebase to figure out what "improve error handling" means. Better:

```
The OrderService throws generic errors. Add typed error classes
(OrderNotFound, InsufficientInventory, etc.) and update error
responses to include error codes. Start from src/services/OrderService.ts.
```

No exploration needed. Claude Code reads one file, understands the scope, acts.

**Scenario 3 — Long conversation history with unrelated tasks:**

Task 1: Add a payment feature (40 messages back and forth)
Task 2: Refactor logging (35 messages back and forth)
Task 3: Add monitoring (current task, 5 messages so far)

The conversation history from Tasks 1 and 2 is still taking up space. Most of it isn't relevant to monitoring. The context window is narrower for Task 3 than it would be in a fresh session.

> ⚠️ **Counterintuitive:** Starting a fresh session can be *faster* than continuing one, even though you lose conversation history. The efficiency gain from having a full context window for the current task outweighs the one-time cost of re-briefing Claude Code via `/init` or `CLAUDE.md`.

### Pull Prompting: The Mechanics

When you instruct Claude Code to fetch context, you're telling it:
1. What goal to accomplish
2. What files/information are relevant (without pasting them)
3. Let Claude Code decide the read order

```
Task: Add structured logging to the audit trail.

Relevant files:
- src/audit/AuditLog.ts — current audit log model
- src/services/UserService.ts — example of a service that writes audits
- src/utils/logger.ts — existing logger utility

Start by reading AuditLog.ts to understand the current shape,
then read UserService.ts to see how audits are currently created,
then propose a structured logging format.
```

Claude Code will:
1. Read AuditLog.ts (maybe 300 tokens)
2. Read UserService.ts (maybe 400 tokens)
3. Use remaining context for reasoning and proposing (the valuable part)

vs. you pasting both files (~1,500 tokens) and only ~1,500 tokens available for reasoning.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Recognizing Context Saturation

You're six tasks into a session. The current task is: *"Optimize the checkout flow for performance."*

**Signal 1 — Slower responses:** Each response takes noticeably longer to generate (30+ seconds where earlier responses were 5–10 seconds).

**Signal 2 — Shallow reasoning:** Claude Code's explanations are shorter, less detailed. Fewer follow-up questions.

**Signal 3 — Pattern repetition:** It keeps suggesting patterns from earlier tasks that aren't relevant. (E.g., "add a queue" — relevant for Task 2 (async background jobs), completely wrong for checkout performance.)

**Signal 4 — Instructions ignored:** You said "focus on database queries," but Claude Code is optimizing HTTP calls. It's partially ignoring the current instructions because it can't hold them in context relative to all the prior conversation.

**Your decision:**
```
/clear
```

Then re-brief with a fresh instruction:

```
New session. Goal: optimize the POST /checkout endpoint for performance.

Constraints:
- Focus on database query efficiency — not HTTP calls or async
- Use the existing caching strategy in src/caching/OrderCache.ts
- Don't refactor the core checkout logic, only optimize queries
- Run performance tests before/after with npm run perf:checkout

Start by reading src/routes/checkout.ts to see the current flow.
```

Fresh session, full context window, focused task.

### Scenario 2: Rewriting Prompts from Push to Pull

**Before (push — you paste files):**
```
I need to add validation to user creation. Here's the UserService:
[entire UserService.ts — 400 lines, 2,000 tokens pasted]

Here's the User model:
[entire User.ts — 150 lines, 800 tokens pasted]

Now add validation for email format, password strength, and required fields.
```

Context spent: 2,800 tokens before Claude Code even starts reasoning.

**After (pull — Claude Code fetches what it needs):**
```
Add input validation to user creation in src/services/UserService.ts.

Validation rules:
- Email: must match RFC 5322 email format
- Password: minimum 12 characters, at least one uppercase, one number, one symbol
- Required fields: email, password, firstName, lastName

Start by reading src/services/UserService.ts to see the current createUser
method. Then read src/models/User.ts to understand the data model. Then
propose validation and show where to add it.
```

Context spent: ~500 tokens for the instruction. 400 tokens for the files Claude Code actually reads. ~1,100 tokens available for reasoning and proposing a solution.

**Output quality:** The second is better. More context for thinking. Faster response. More detailed reasoning.

### Scenario 3: Three Vague Prompts → Precise Versions

**Prompt 1 (vague):**
```
Refactor the payment module.
```

Claude Code must guess: what's wrong? What's the goal? How much scope? It wastes context exploring before it can act.

**Prompt 1 (precise):**
```
Refactor the payment module to separate Stripe-specific code from the
business logic. Create an IPaymentProvider interface. Implement
StripePaymentProvider that handles SDK calls. Update PaymentService
to depend on the interface, not the Stripe SDK directly. This will
let us add Azure Payment Services or another provider later without
touching PaymentService.

Start from src/services/PaymentService.ts. Show the plan first
before making changes.
```

Claude Code knows exactly what to do. No exploration. Lower context burn.

---

**Prompt 2 (vague):**
```
Improve error handling.
```

**Prompt 2 (precise):**
```
Currently, errors from the payment service are thrown as generic Error
objects with messages like "Payment failed." Add typed error classes:
- PaymentError (base)
- PaymentRejectedError (card declined, etc.) — should be 400
- PaymentGatewayError (Stripe API error) — should be 500
- PaymentTimeoutError (slow Stripe response) — should be 503

Update src/services/PaymentService.ts and the error handler in
src/middleware/errorHandler.ts to use the new classes. Include
HTTP status codes in the error definitions. Propose the error
class definitions first, then update the service.
```

Precise scope, clear success criteria, minimal context burn.

---

**Prompt 3 (vague):**
```
Add tests to this module.
```

**Prompt 3 (precise):**
```
Add unit tests for src/utils/dateUtils.ts using Jest. Cover:
- parseISO8601() — valid date, invalid format, null input, timezone handling
- addDays() — adding positive/negative days, leap year edge cases
- formatAsISO8601() — round-trip (parse then format should equal input)

Exclude integration tests — those live in a separate test run. Place
tests in src/utils/__tests__/dateUtils.test.ts. Use the test fixtures
at src/test-utils/fixtures.ts for sample data.
```

Scope is bounded. Test framework is named. File locations are specified. Success is measurable.

## 6. Practical Usage & Decision Guidance

### When to /clear (Start Fresh)

| Signal | Decision |
|---|---|
| Current task is unrelated to all prior tasks | `/clear` |
| Conversation history exceeds 50 exchanges | `/clear` (unless high continuity value) |
| Claude Code's responses have become noticeably slower (30+ sec) | `/clear` |
| Output quality has degraded (shallow reasoning, repetition) | `/clear` |
| You need to switch teams/projects | `/clear` |
| Prior tasks were exploratory; current task is focused | `/clear` |
| Current task will require many back-and-forths (20+) | Consider `/clear` to preserve context |

### Context Budgeting for Multi-Task Sessions

**Rule of thumb:** Allocate context roughly proportionally to task complexity.

| Task type | Est. context | Context budget |
|---|---|---|
| Simple (single file edit) | 20% | 2,000 tokens |
| Medium (multi-file feature) | 40% | 4,000 tokens |
| Complex (architectural change) | 50%+ | 5,000+ tokens |

If a single task is consuming more than its budget, use `/clear` between tasks rather than letting context accumulate.

### Prompt Precision Checklist

Before submitting a task, verify:

- [ ] Goal is stated in one sentence
- [ ] Constraints are specific (not "be careful," but "maximum 5ms latency")
- [ ] Files involved are named explicitly
- [ ] Success criteria are measurable ("all tests pass," not "it works")
- [ ] Any non-obvious context is provided (e.g., "this codebase uses custom error base classes in `src/errors/`")
- [ ] You haven't pasted files that Claude Code can read itself

## 7. Common Pitfalls & Misconceptions

**"The bigger the context window, the better."**
A bigger window helps, but poor context management wastes even a large one. An engineer who uses 80% of a large window efficiently is better off than one who fills it with irrelevant files. The skill is selectivity, not size.

**"I'll paste everything to be safe."**
The reality: pasting unnecessary files makes Claude Code slower and less accurate. You're consuming tokens that could go to reasoning. The vague instruction "here's my whole codebase, figure it out" burns more context than a precise one that names exactly what's needed.

**"Conversation history is always useful — I'll keep the session open."**
After 3–4 tasks, conversation history becomes more noise than signal. If tasks are unrelated, history confuses more than it helps. The value of continuity expires. When in doubt, `/clear` if the next task is dissimilar.

**"If I use pull prompting, I have to manually list every file."**
You don't need to list every file — just the entry points and boundaries. "Start from src/services/OrderService.ts and read all its imports" tells Claude Code to follow the dependency chain itself. Trust it to explore efficiently.

**"A vague prompt is flexible — it lets Claude Code figure out the best approach."**
Vague prompts waste context and produce mediocre results. Claude Code doesn't "figure out the best approach" — it wastes tokens guessing what you meant. Precision always wins.

## 8. Trade-offs, Scale, and Edge Cases

Pull prompting requires Claude Code to make reads, which adds latency — each file read is a round-trip. For very large tasks where you know exactly which files are needed and won't change, push prompting (pasting) is slightly faster. But the speed trade-off is usually worth the context efficiency gain.

Context saturation isn't a cliff — it's gradual degradation. As you approach limits, output quality doesn't drop to zero; it slowly declines. You might not notice until you're well into the saturated region. The disciplines of explicit `/clear` decisions and pull prompting prevent that slow degradation.

For teams: `/clear` between team members using a shared session is a good habit. Each person's work has context footprints. Starting fresh when a new engineer picks up work reduces confusion and context collisions.

## 9. Self-Check Questions

1. You're in a session with a full conversation history. The next task is completely unrelated to prior work. The context window still has room (you haven't hit the limit). Do you `/clear` or continue? What's your reasoning?

2. You're about to ask Claude Code to "improve the API performance." Rewrite this as a precise prompt that manages context efficiently.

3. A colleague pastes a 100-line file and says: "Here's the whole thing, please optimize it." What's a more context-efficient way to frame this request, and why does it matter?

4. You're five tasks into a session. Responses are now 30+ seconds each (they were 5–10 seconds earlier). Reasoning is becoming vague. What's happening, and what's the fix?

5. You're writing a prompt with a list of five files the task needs. Instead of pasting them, you want Claude Code to read them. Write the prompt framing that makes this work.

## 10. What to Learn Next

- **CLAUDE.md & Project Memory** — Persistent context loaded automatically at session start reduces the need to re-brief Claude Code. Combining precise prompting with a good `CLAUDE.md` is the compounding practice.
- **Multi-Agent Workflows** — When a task is too large for one session (you'd need to `/clear` multiple times), parallel agents with scoped context each handle parts independently. Single-session context management is the prerequisite.
- **Output Quality Auditing** — Sessions where context is managed poorly produce lower-quality output that gets flagged in audits. The audit findings (generic errors, missed patterns) often trace back to context saturation in the original session.

## Summary

Context is the scarcest resource in a Claude Code session. Managing it well means distinguishing between continuity value and accumulated noise — using `/clear` when history stops helping. It means writing precise prompts that tell Claude Code *what it needs* rather than you pasting everything you think it might need. It means recognizing saturation (slow responses, shallow reasoning, pattern repetition) as a signal to start fresh. The engineers who do more per session aren't the ones with the biggest context windows — they're the ones who waste the least context on irrelevance.
