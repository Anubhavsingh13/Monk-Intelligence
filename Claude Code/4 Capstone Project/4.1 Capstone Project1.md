## What This Page Is

Every earlier wiki page taught a skill in isolation. This page connects them. You'll follow a single realistic project — an Azure-hosted order management API — through five scenarios that span an entire sprint. Each scenario pulls from multiple wiki pages, cross-referenced explicitly so you can see which skills fire in which situations and how they chain together.

Think of this as a worked final exam: no new theory, just applied judgement.

## The Project: OrderFlow API

**Stack:** Node.js 20, TypeScript, Azure Functions v4, Cosmos DB, Azure Service Bus, Application Insights
**Team size:** 3 engineers
**State at sprint start:** Working product, partial test coverage (38%), 312 ESLint warnings, no `CLAUDE.md`, one open P1 bug, one feature request from product

This is a realistic starting point — not a greenfield project, not a legacy disaster. The kind of codebase most engineers actually work in.

## Scenario 1: Day 1 — Onboarding Claude Code to an Existing Project

**Skills used:**
- Page 1: Installation & first run
- Page 2: Navigation & permissions
- Page 8: CLAUDE.md basics
- Page 9: CLAUDE.md scoping

**The situation:** The team has decided to adopt Claude Code. No one has used it on this codebase before. Your first task: set up the tooling and create the project memory that every future session will build on.

---

### Step 1.1: Install and Configure

Install the CLI and authenticate (Page 1):

```bash
npm install -g @anthropic-ai/claude-code
claude --version
```

Navigate to the project root and start your first session:

```bash
cd ~/projects/orderflow-api
claude
```

### Step 1.2: Explore Before You Configure

Before writing `CLAUDE.md`, understand the codebase (Page 2 + Page 4):

```
I'm new to this codebase. Starting from src/index.ts, give me a map:
what are the main modules, what does each one own, and how does a
POST /orders request flow from the Azure Function trigger to Cosmos DB?
```

Claude Code reads the entry point, follows imports, traces the call chain. It produces:

```
Request flow: HTTP trigger (src/functions/createOrder.ts)
  → OrderController (src/controllers/OrderController.ts)
  → OrderService (src/services/OrderService.ts)
  → OrderRepository (src/repositories/OrderRepository.ts) → Cosmos DB
  → EventPublisher (src/events/OrderEventPublisher.ts) → Service Bus
```

Follow up — surface the hidden coupling (Page 4):

```
Does OrderService have any side effects that aren't visible from its
imports? Check for event publishing, queue writes, or shared state.
```

Claude Code finds that `OrderService` calls `EventPublisher` to emit an `OrderCreated` event to Service Bus — a critical side effect not obvious from the function signature.

### Step 1.3: Run /init and Edit

Generate the scaffold (Page 8):

```
/init
```

Claude Code produces a basic `CLAUDE.md` with tech stack, directory structure, and commands. Now apply the inference test from Page 9 — strip out everything Claude Code can already infer:

**Remove:**
- "This is a TypeScript project" (inferrable from `tsconfig.json`)
- "We use Jest" (inferrable from `jest.config.ts`)
- "Package manager: npm" (inferrable from `package-lock.json`)

**Add by hand** (Page 8 — the four high-impact categories):

```markdown
# CLAUDE.md — OrderFlow API

## Architecture
- Functions → Controllers → Services → Repositories. Never skip a layer.
- All secrets from Key Vault via src/config/secrets.ts. Never use process.env directly.
- OrderService emits events to Service Bus via EventPublisher. This side effect
  is critical — never remove or bypass it.

## Coding Conventions
- Logger: src/utils/logger.ts. Never console.log or console.error.
- Errors: always use AppError subclasses from src/errors/. Never throw generic Error.
- Azure Functions: v4 programming model (app.http(), app.serviceBusTopic()).

## Testing Policy
- Regression test required for every bug fix
- New service methods: minimum one happy path + one error path test
- No tests that only assert .toBeDefined()

## Forbidden Patterns
- No console.* — use the app logger
- No direct Cosmos DB queries outside repository files
- No new npm packages without noting the reason
```

### Step 1.4: Add a Child CLAUDE.md for Payments

The payments module has strict rules (Page 9):

```markdown
# src/payments/CLAUDE.md

## Invariants
- All monetary amounts in pence (integer). Never float.
- All charge operations require an idempotency key.
- Never call Stripe SDK directly — use PaymentGateway interface.

## Error Handling
- All errors: PaymentError with transactionId. Never generic Error.
```

**Pages applied in this scenario:** 1, 2, 4, 8, 9
**Time spent:** ~90 minutes
**Outcome:** Claude Code is installed, configured, and has persistent project memory. Every future session starts informed.

## Scenario 2: Day 2 — Fixing the P1 Bug

**Skills used:**
- Page 3: Modes (Plan mode for investigation)
- Page 4: Bug fixing workflow
- Page 5: Root cause analysis
- Page 5: Writing regression tests
- Page 10: Context management

**The situation:** The P1 bug: *"Orders with digital-only products fail to process. Error: TypeError: Cannot read properties of undefined (reading 'street') at OrderService.ts:142."*

---

### Step 2.1: Investigate in Plan Mode

Start a fresh session (Page 10 — this is unrelated to the setup work):

```bash
claude
```

Submit the bug with the three-layer model in mind (Page 4 — symptom, trigger, root cause):

```
P1 bug investigation. Do not fix anything yet.

Symptom: Orders with digital-only products fail to process.
Error: TypeError: Cannot read properties of undefined (reading 'street')
  at OrderService.ts:142

Trigger: Occurs only when the order contains no physical items.

Start from src/services/OrderService.ts line 142. Trace what's accessed,
where it comes from, and what condition makes it undefined. I want the
root cause, not a quick fix.
```

Claude Code reads `OrderService.ts:142`, finds `order.shippingAddress.street` — accessed unconditionally. Digital-only orders have `shippingAddress: null` because no shipping is needed. The code assumes every order has a shipping address.

### Step 2.2: Perform Root Cause Analysis

Push past the immediate fix (Page 5 — RCA):

```
Before we fix this, I want to understand the pattern.

1. What assumption was violated?
2. Search the codebase for other places where order.shippingAddress
   is accessed without a null check — especially in services, controllers,
   and event publishers.
3. Are there other optional order fields (billingAddress, discountCode,
   giftMessage) that have the same access pattern?

Give me the full audit before we fix anything.
```

Claude Code finds:
- `order.shippingAddress` accessed without null check in **4 locations**
- `order.billingAddress` accessed without null check in **2 locations**
- `order.discountCode` accessed without null check in **1 location**

**This is no longer a single bug fix — it's a systemic pattern.** You now have a design conversation for the team: add a validation layer at the service boundary.

### Step 2.3: Fix and Write Regression Tests

Fix the immediate P1 (still in the same session — context is relevant):

```
Fix the null access on order.shippingAddress in OrderService.ts line 142.
The fix should check whether shippingAddress exists before accessing
its properties. For digital-only orders, skip the shipping address
formatting entirely.

Then write a regression test per our CLAUDE.md policy:
- File: src/services/__tests__/OrderService.test.ts
- Test name: "should process order with no shipping address (digital-only)"
- Must fail on the unfixed code, pass on the fixed code
- Also add: "should process order with full shipping address (physical items)"
```

### Step 2.4: Log the Pattern for the Sprint

The other 6 locations are not P1 — but they're the same root cause. Create a ticket (or note for the team) with the RCA audit output: pattern, locations, recommended fix (service-boundary validation).

**Pages applied in this scenario:** 3, 4, 5, 10
**Time spent:** ~45 minutes (investigation + fix + regression test + audit)
**Outcome:** P1 fixed with regression test. 6 additional vulnerabilities identified. Systemic pattern documented for the team.

## Scenario 3: Day 3–4 — Building the Feature

**Skills used:**
- Page 3: Modes (Plan mode for design, step-by-step execution)
- Page 6: Feature building (Tier 3 — new architectural component)
- Page 6: Modularity test
- Page 10: Context management (/clear between phases)

**The situation:** Product has requested: *"Add a notification system. When an order is confirmed, send an email to the customer. We'll add SMS and in-app notifications later."*

This is a Tier 3 feature — it requires a new architectural component that future channels will depend on.

---

### Step 3.1: Phase 1 — Design the Contract (Plan Mode)

```bash
claude --plan
```

```
Design a pluggable notification system. Do NOT write implementation code.

Produce only:
1. INotificationProvider interface
2. NotificationPayload type (channel-agnostic)
3. NotificationRouter — routes a payload to registered providers
4. How a new channel (e.g., SMS) would be added in future — what files
   does a developer create, what existing files change?

Constraints:
- Adding a new channel must not require modifying NotificationRouter
  or any existing provider
- Providers must be registered at startup, not hardcoded
- Must support multiple channels per event (email + in-app for confirmations)
- Call sites use notificationService.send(payload) with no channel knowledge
```

Review the interfaces. Before approving, run the modularity test (Page 6):

```
Simulate adding a Microsoft Teams notification channel.
What files would be created? What existing files would change?
If more than 2 existing files change, explain the coupling and fix it.
```

Claude Code confirms: one new file (`TeamsNotificationProvider.ts`), one registration line at startup. No existing providers or router change. Design is clean.

### Step 3.2: /clear and Phase 2 — Implement Step by Step

The Phase 1 conversation has served its purpose. Start fresh for implementation (Page 10):

```
/clear
```

New session with a focused implementation instruction:

```
Implement the notification system designed in the approved plan.
The interfaces are:
- INotificationProvider (send method, channel identifier)
- NotificationPayload (recipientId, templateId, data, channels[])
- NotificationRouter (register providers, route by channel)

Execute step by step. Stop after each step and wait for my approval.

Step 1: Create interface files in src/notifications/
Step 2: Implement EmailNotificationProvider using @sendgrid/mail
Step 3: Implement NotificationRouter with provider registration
Step 4: Wire into app startup in src/app.ts
Step 5: Update OrderService to call notificationService.send()
  after order confirmation
Step 6: Write unit tests for each component
```

Each step produces a reviewable diff. You approve step by step (Page 6 — "stop after each step" is the mechanism that keeps a 15-file change reviewable).

### Step 3.3: Validate the Feature

After all steps are approved, run the full suite:

```
Run npm test and npm run lint. Report results.
If any tests fail, diagnose and fix without changing the public
interfaces we just implemented.
```

**Pages applied in this scenario:** 3, 6, 10
**Time spent:** ~2.5 hours across two sessions (design + implementation)
**Outcome:** Pluggable notification system with email provider. SMS and in-app can be added later with one new file each. No existing code changes needed.

## Scenario 4: Day 5 — Technical Debt Cleanup

**Skills used:**
- Page 5: Lint fixing at scale
- Page 7: Refactoring workflows
- Page 7: Dependency upgrade
- Page 9: CLAUDE.md audit

**The situation:** Sprint buffer day. The team decides to tackle: 312 ESLint warnings, a messy module, and an overdue SDK upgrade.

---

### Step 4.1: Lint Blitz (Morning — 1 Hour)

Fresh session (Page 10 — isolated task):

```
This codebase has 312 ESLint warnings. Fix them all in this session.

Step 1: Run npx eslint --fix src/ to apply all auto-fixable rules.
  Report how many remain.

Step 2: Group remaining warnings by rule name and count. Show the top 5.

Step 3: For the top 5:
  - Consistent style issue → fix all instances
  - Requires judgement → show me 2 examples, ask how to handle
  - False positive for our codebase → add targeted suppression with comment

Do not fix anything in Steps 2–3 until I approve the strategy.
```

Result: 198 auto-fixed. 114 remain. Top 5 rules addressed with a strategy. 40 minutes. Green CI pipeline.

### Step 4.2: Refactor the Messy Module (Midday — 1 Hour)

`/clear` — new task (Page 10).

`OrderProcessor.ts` is 380 lines with four interleaved concerns (Page 7):

```
Refactor src/orders/OrderProcessor.ts.

Goal: extract each concern into a named private method so processOrder()
reads as a sequence of high-level steps.

Constraints:
- Do NOT change any logic — only restructure
- Do NOT change the public interface
- Do NOT fix bugs you spot — note them separately
- Do NOT move code to new files in this step

After the refactor, processOrder() should read like:
  validateOrder(order)
  reserveInventory(order)
  processPayment(order)
  sendConfirmation(order)

Produce the plan first.
```

Review the plan. Verify extracted method signatures. Approve. Review the diff with the Page 7 checklist:

- [ ] Does processOrder() read as a sequence of named steps?
- [ ] Are all extracted methods private?
- [ ] Is any logic missing?
- [ ] Did error handling change position?
- [ ] Do all tests pass without modification?

### Step 4.3: Dependency Upgrade (Afternoon — 1 Hour)

`/clear` — new task (Page 10).

Upgrade `@azure/cosmos` from v3 to v4 (Page 7 — three-phase model):

```
Upgrade @azure/cosmos from v3 to v4.

Phase 1: Before touching code, read the changelog and summarise:
- Every breaking change in the version range
- Every deprecated method we might be using
- Any semantic changes (behaviour changes without code changes)

Do not update code until I review the summary.
```

Review the summary. Note that `Container.items.query()` now returns a different iterator pattern, and `ConflictResolutionPolicy` has changed defaults.

```
Phase 2: Update all call sites in src/repositories/. For each change,
show before/after and explain why the change is required.

Phase 3: Run npm test and npm run test:integration. Report results
and diagnose any failures.
```

Two integration tests fail — the iterator pattern change requires updating how results are consumed in `OrderRepository.ts`. Claude Code diagnoses and fixes. All tests pass.

### Step 4.4: Update CLAUDE.md from the Day's Work

End-of-day audit (Page 9 + Page 7):

```
Based on today's session, what corrections or patterns should be
added to CLAUDE.md? Check:
- Did I correct Claude Code on anything it should know permanently?
- Did the lint blitz reveal any conventions worth encoding?
- Did the refactor reveal invariants worth documenting?
```

Additions to `CLAUDE.md`:

```markdown
## Conventions (added Day 5)
- ESLint suppressions must include a comment explaining why.
  No bare // eslint-disable-next-line without a reason.

## Architecture (added Day 5)
- OrderProcessor.processOrder() is a coordination method — it calls
  validateOrder, reserveInventory, processPayment, sendConfirmation
  in sequence. Do not add logic directly to processOrder.
- @azure/cosmos v4 — uses async iterators for query results.
  See OrderRepository.ts for the pattern.
```

**Pages applied in this scenario:** 5, 7, 9, 10
**Time spent:** ~3 hours across three focused sessions
**Outcome:** 312 lint warnings → 0. Messy module refactored. SDK upgraded. `CLAUDE.md` updated with three new standing instructions.

## Scenario 5: Day 6–7 — Sprint Review, Output Audit & Handoff

**Skills used:**
- Page 7: Weekly output audit
- Page 9: CLAUDE.md quarterly audit
- Page 5: Incremental regression strategy

**The situation:** Sprint is ending. Time to review what Claude Code produced this week, identify patterns, and prepare the project for the next sprint.

---

### Step 5.1: Weekly Output Audit

Review all Claude Code output from the sprint (Page 7):

```
Audit the code changes made this sprint. Review the git log for
the past 5 days.

For each change, identify:
1. Changes that needed manual correction — what instruction would
   have prevented it?
2. Patterns that were consistently good — what produced that quality?
3. Anything I approved that, on reflection, I'd do differently?

Output format:
- Corrections needed: [pattern] → [standing instruction for CLAUDE.md]
- Quality patterns: [what worked]
- Retrospective concerns: [second-look items]
```

**Audit findings:**

| Finding | Type | Action |
|---|---|---|
| Claude Code used `any` type in 2 test files | Correction | Add to CLAUDE.md: "No `any` in test files either — use proper types or `unknown`" |
| Notification interfaces were clean on first attempt | Quality | The Phase 1 contract-first approach (Page 6) consistently produced cleaner output |
| Error messages in new code inconsistent with existing style | Correction | Add to CLAUDE.md: "Error messages: sentence case, no period, include entity ID" |
| The refactored OrderProcessor reads well | Quality | The "do NOT fix bugs during refactor" constraint worked — keep using it |
| One regression test only asserts `.toBeDefined()` | Retrospective | Rewrite it to assert specific return shape. Testing policy in CLAUDE.md wasn't specific enough |

### Step 5.2: Update CLAUDE.md and Audit It

Apply the audit findings, then run the CLAUDE.md audit from Page 9:

```
Read the CLAUDE.md. For each instruction, evaluate:
1. Inferrable from codebase? → Remove
2. Too vague? → Rewrite or remove
3. Outdated? → Update or remove
4. Wrong scope? → Move to child CLAUDE.md
5. Contradictory? → Resolve

Output a table: instruction | category | action
```

The audit trims 3 inferrable lines added during `/init`, sharpens 2 vague instructions, and moves 1 payment-specific rule to the child file.

### Step 5.3: Assess Regression Test Coverage

Check the sprint's contribution to the regression pack (Page 5):

```
List all test files modified or created this sprint. For each:
1. What bug or feature does it protect?
2. Would it catch a regression if the protected code changed?
3. Does it follow our testing policy (no .toBeDefined()-only assertions)?

Report any gaps.
```

**Sprint test summary:**

| Test | Protects | Quality |
|---|---|---|
| OrderService — digital-only order | P1 bug (null shippingAddress) | Good — asserts specific behaviour |
| EmailNotificationProvider | New feature (email sending) | Good — tests send + failure paths |
| NotificationRouter | New feature (routing logic) | Good — tests registration + multi-channel |
| OrderProcessor — refactored methods | Refactoring correctness | Weak — one test only asserts `.toBeDefined()` → rewrite |

One test needs rewriting. The regression pack grew by 11 tests this sprint. Coverage moved from 38% to 52%.

## Key Takeaways — What This Case Study Demonstrates

1. **Skills don't operate in isolation.** Every real task pulls from 3–5 wiki pages simultaneously. The curriculum teaches skills separately; production uses them together.

2. **`/clear` is a productivity tool, not a last resort.** This sprint used 8 sessions, not 1 long one. Each `/clear` gave the next task a full context window. The compounding loss from context saturation is worse than the one-time cost of re-briefing.

3. **`CLAUDE.md` is a living document.** It was created on Day 1, updated on Day 5, and audited on Day 7. It grew where needed, shrank where inferrable, and moved content to child files where scope demanded. A `CLAUDE.md` that isn't updated is a `CLAUDE.md` that stops being useful.

4. **The output audit closes the loop.** Without the Day 7 audit, the weak test assertion, the inconsistent error messages, and the `any` types in tests would have compounded into standing patterns. The audit caught them and converted them to standing instructions.

5. **Modularity decisions pay forward.** The Phase 1 contract-first design for the notification system means the next sprint's "add SMS" feature is a single new file — not a rewrite. That 30 minutes of design work saved an estimated 4 hours of future work.

## Self-Assessment

After reading this case study, you should be able to answer:

1. On Day 2, why did we ask Claude Code "don't fix anything yet" before investigating? What would have gone wrong if we hadn't?

2. On Day 3, why did we `/clear` between Phase 1 (design) and Phase 2 (implementation)? What does this tell you about context management?

3. On Day 4, why did we separate lint fixing, refactoring, and the dependency upgrade into three separate sessions? Why not do all three in one?

4. On Day 5, the output audit found a weak test assertion that was approved during Day 2. What does this say about the limitation of in-session review?

5. If this sprint's `CLAUDE.md` updates carry forward to the next sprint, which specific corrections will the team *not* have to make again?
