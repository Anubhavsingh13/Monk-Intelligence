# Claude Code: Refactoring, Dependency Upgrades & Output Quality Review

**Last Updated:** 2026-04-15

> *The mark of a mature Claude Code user isn't how fast they ship — it's how well they review what was shipped, and how systematically they improve the process.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Direct Claude Code to safely refactor a messy module — extract functions, rename symbols, restructure logic — while preserving behaviour
- Read a refactoring diff critically: know what signals indicate a safe change vs. a behaviour-altering one
- Ask Claude Code to perform a dependency upgrade, handle breaking changes, and validate the result
- Run a structured weekly output audit: identify what Claude Code produced well, what needed correction, and what that pattern tells you about your prompting
- Adjust your instructions and `CLAUDE.md` based on audit findings — turning one-off corrections into permanent improvements
- Recognise the class of changes where Claude Code should not be trusted without human verification

---

## 1. Why This Matters (In Our Systems)

Three things happen when teams get comfortable with Claude Code and stop reviewing output carefully:

**First:** A refactor is accepted because the tests pass and the diff looks clean. Three weeks later, a subtle behaviour change — a changed evaluation order, a moved side effect, a renamed symbol that broke a dynamic import — surfaces in production. The tests didn't catch it because they weren't written for that behaviour. The review didn't catch it because the diff was large and the reviewer trusted the tool.

**Second:** A dependency upgrade is applied. Claude Code updated 14 call sites. It missed the 15th — a utility function called only in one integration test. The upgrade ships. The integration fails in staging, not locally, because the test environment uses the real external service.

**Third:** No one audits what Claude Code has been producing. The team doesn't notice that Claude Code consistently generates over-verbose error handling in this codebase, or that it keeps reaching for a deprecated SDK method because no one told it the preferred alternative. The same correction is made manually, repeatedly, in every PR.

All three are solved by the same discipline: structured review, not just diff approval.

## 2. Intuition & Mental Models

### Refactoring: Behaviour Preservation as the Invariant

Refactoring has a precise definition: *changing the structure of code without changing its observable behaviour.* Everything else — changing what the code does, fixing a bug in the process, adding a parameter — is not refactoring. It's a combined change, and combined changes are harder to review and harder to revert.

When you ask Claude Code to refactor, you're asking it to honour this invariant. The way you verify it did: the tests pass, and every public interface that existed before still exists after with the same signature. If either is false, the "refactor" changed behaviour.

The mental model: a good refactoring diff should look like a *reorganisation*, not a *rewrite*. The same logic, in a more readable home.

### Dependency Upgrades: Trust the Changelog, Not the Diff

When Claude Code updates a dependency, it reads the new API and rewrites your call sites. What it cannot do: read the changelog for subtle semantic changes — things that compile and run but behave differently (e.g., a method that used to be synchronous is now asynchronous, or a default option that changed value between versions).

The mental model: Claude Code handles syntax, you handle semantics. The upgrade is complete when both are verified.

### Output Auditing: The Feedback Loop You Own

Claude Code improves its output in your codebase in two ways: you correct it in the session, and you encode those corrections in `CLAUDE.md` for all future sessions. The first is reactive. The second is compounding.

A weekly audit is the mechanism that converts reactive corrections into compounding improvements. Without it, you correct the same mistake ten times. With it, you correct it once and the correction becomes a standing instruction.

## 3. Core Concepts & Terminology

**Behaviour-preserving refactoring** — A code change that alters structure (naming, organisation, abstraction level) without changing observable output or side effects. The gold standard: tests pass before and after, public interfaces are unchanged.

**Mechanical refactoring** — A refactoring that can be applied as a deterministic transformation: rename all usages of `X` to `Y`, extract lines 40–60 into a named function, inline a single-use variable. These are the safest class of refactoring and the best fit for Claude Code.

**Semantic refactoring** — A refactoring that requires understanding the *meaning* of the code to restructure it correctly: reorganising a state machine, inverting a control flow, decomposing a god object. Higher judgement required; more review needed.

**Breaking change** — A change in a dependency's API that requires call sites to be updated. Can be syntactic (renamed method, changed signature) or semantic (changed behaviour, changed default).

**Migration guide** — A document published by a dependency's maintainers describing every breaking change between major versions and how to update call sites. Claude Code should read this before upgrading, if you point it at the right resource.

**Output audit** — A structured review of a batch of Claude Code's output — typically a week's worth of PRs or sessions — looking for patterns: what was consistently good, what needed consistent correction, and what those patterns imply about instruction quality.

**Standing instruction** — A directive in `CLAUDE.md` that applies to every session, without requiring re-entry. The product of a good output audit: corrections that are worth making permanent.

## 4. How It Works (What Actually Matters)

### Refactoring: What Claude Code Does Well and Doesn't

Claude Code is excellent at mechanical refactoring: renaming symbols consistently across files, extracting a block of code into a named function, removing duplication by introducing a shared utility, flattening deep nesting by inverting conditions. These transformations are deterministic, testable, and reviewable.

It is less reliable at semantic refactoring without explicit guidance: restructuring a complex algorithm, decomposing a class that has accreted responsibilities over years, or reorganising module boundaries. For these, use Plan mode and review the proposed structure before any code is written.

> ⚠️ **Counterintuitive:** The more "obvious" a refactoring feels, the more dangerous it is to skip the review. Renaming a method that has 40 call sites feels mechanical. But if one of those call sites uses the method name as a string for dynamic dispatch, the rename will silently break it. Claude Code will update the 40 explicit call sites and miss the string reference. Always grep for string usage of renamed symbols after a rename refactoring.

### Dependency Upgrades: The Three-Phase Model

Safe dependency upgrades follow three phases regardless of tool:

1. **Understand the breaking changes** — read the migration guide or changelog for the version range you're crossing
2. **Update call sites** — apply the syntactic changes the new API requires
3. **Validate semantics** — run tests, check runtime behaviour for any silent semantic changes

Claude Code handles phase 2 well. It needs your help with phase 1 (point it at the migration guide) and cannot fully automate phase 3 (you run the tests and verify runtime behaviour).

> ⚠️ **Counterintuitive:** A dependency upgrade where all tests pass is not necessarily complete. Tests verify the behaviours they cover. If a semantic change affects an untested code path — or a behaviour that tests don't explicitly assert — passing tests give false confidence. Check the migration guide for semantic changes even after tests pass.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Refactoring a Messy Module

**The situation:** `src/orders/OrderProcessor.ts` is 420 lines. It contains order validation, payment processing, inventory reservation, and email dispatch — all in one `processOrder()` function. You want it broken up without changing behaviour.

**Step 1 — Describe the refactoring goal precisely:**

```
Refactor src/orders/OrderProcessor.ts. The processOrder() function is 
420 lines and handles four concerns: validation, payment, inventory, 
and notification.

Goal: extract each concern into a named private method, so processOrder() 
reads as a sequence of high-level steps.

Constraints:
- Do NOT change any logic — only restructure
- Do NOT change the public interface of OrderProcessor
- Do NOT fix any bugs you spot in the process — note them separately
- Do NOT move code to new files in this step — only extract to private methods
  within the same class

After the refactor, processOrder() should read like:
  validateOrder(order)
  reserveInventory(order)
  processPayment(order)
  sendConfirmation(order)

Produce the plan first. Show me the proposed method signatures and 
what lines each will contain before writing any code.
```

The "do NOT fix bugs" constraint is critical. Without it, Claude Code will helpfully fix things it spots — producing a combined refactor+bugfix that is much harder to review and impossible to safely revert independently.

**Step 2 — Review the diff with this checklist:**

- [ ] Does `processOrder()` now read as a sequence of named steps?
- [ ] Are all extracted methods private (not part of the public interface)?
- [ ] Are the method signatures consistent with what the plan proposed?
- [ ] Is any logic missing from the extracted methods vs. the original?
- [ ] Did any error handling move in a way that changes when errors are caught?
- [ ] Do all tests still pass without modification?
- [ ] Are there any renamed variables that might affect external references?

The last two points are the ones most likely to reveal a behaviour change disguised as a refactor.

### Scenario 2: Refactoring by Renaming — The Hidden Risk

**The task:** rename `UserService.getUser()` to `UserService.findById()` across the codebase for consistency.

```
Rename UserService.getUser() to UserService.findById() across the 
entire codebase. Update all call sites.

After completing the rename, also grep for the string "getUser" in 
case it's used in any string literals, dynamic calls, or test 
descriptions. Report any string occurrences separately — do not 
change them automatically.
```

The grep step is the safety net. In JavaScript/TypeScript codebases, method names frequently appear as strings in:
- Test `describe`/`it` block names (*not a bug, but worth knowing*)
- Dynamic property access: `service['getUser'](id)`
- Reflection, decorators, or OpenAPI generated stubs
- Log messages that include method names for tracing

Without the grep step, these are silent misses. Claude Code will not look for them unless asked.

### Scenario 3: Dependency Upgrade with Breaking Changes

**The task:** upgrade `@azure/service-bus` from v7 to the latest version.

**Step 1 — Give Claude Code the migration material:**

```
Upgrade @azure/service-bus from v7 to v7.9 (latest) in this project.

Before touching any code, read the CHANGELOG for the version range 
we're crossing:
https://github.com/Azure/azure-sdk-for-js/blob/main/sdk/servicebus/service-bus/CHANGELOG.md

Summarise:
1. Every breaking change in the version range
2. Every deprecated method we might be using
3. Any semantic changes (behaviour changes that don't require code changes 
   but affect runtime behaviour)

Do not update any code until I've reviewed the summary.
```

**Step 2 — After reviewing the summary, scope the update:**

```
Based on the breaking changes you found, update all call sites in src/.

For each change:
- Show me the before and after
- Explain why the change is required (not just what changed)

After all call sites are updated, run the test suite and report 
which tests pass, which fail, and your hypothesis for any failures.
```

The "explain why" instruction is not just for learning. It forces Claude Code to engage with the semantics of the change rather than pattern-matching syntax. A Claude Code that can explain why `receiveMessages()` now requires an `abortSignal` is more likely to have applied the change correctly than one that just updated the method signature.

### Scenario 4: Weekly Output Quality Audit

This is the highest-leverage 30 minutes of your week with Claude Code.

**The audit prompt (run on your last week's PRs):**

```
I want to audit the code Claude Code generated this week to improve 
my prompting and CLAUDE.md.

Review the following PRs/diffs: [list git log --oneline for the week, 
or paste relevant PR numbers]

For each, identify:
1. Changes that needed manual correction after Claude Code produced them
   — what was wrong, and what instruction would have prevented it?
2. Patterns that were consistently good — what instruction or context 
   produced that quality?
3. Anything Claude Code generated that I approved but that, on reflection, 
   I'd now do differently

Output format:
- Corrections needed: [pattern] → [standing instruction to add to CLAUDE.md]
- Quality patterns to keep: [what produced them]
- Retrospective concerns: [code that passed review but warrants a second look]
```

**Example audit findings and their conversions to standing instructions:**

| Audit finding | Standing instruction added to `CLAUDE.md` |
|---|---|
| Claude Code kept using `console.error` instead of the app logger | *"Never use `console.log` or `console.error`. Use the logger at `src/utils/logger.ts`."* |
| Error messages were inconsistent — sometimes with codes, sometimes without | *"All error responses must include an `errorCode` field. See `src/errors/codes.ts`."* |
| Tests used `.toEqual()` for object comparison but team standard is `.toMatchObject()` | *"In tests, prefer `.toMatchObject()` over `.toEqual()` for object assertions."* |
| Claude Code added JSDoc to functions it didn't create | *"Do not add JSDoc or inline comments to functions you did not modify."* |
| Azure Function triggers were using `httpTrigger` v3 syntax instead of v4 | *"This project uses Azure Functions v4 runtime. Use the v4 programming model syntax."* |

Each finding is one correction made once. Without the audit, each is a correction made every time.

## 6. Practical Usage & Decision Guidance

### Refactoring: Which Tasks to Delegate, Which to Own

| Refactoring task | Delegate to Claude Code? | Why |
|---|---|---|
| Rename symbol across codebase | Yes, with grep follow-up | Mechanical, but requires string check |
| Extract private methods from long function | Yes, with plan review | Low risk if public interface unchanged |
| Flatten deeply nested conditionals | Yes | Structural, testable |
| Decompose a god class into multiple classes | Plan only first | High semantic risk, needs human design |
| Reorganise module boundaries | Human design + Claude implementation | Architecture decision, not mechanics |
| Inline single-use variables | Yes | Trivially mechanical |
| Convert callbacks to async/await | Yes, with test validation | Generally safe, but async semantics differ |

### Dependency Upgrade Risk by Package Type

| Package type | Risk level | Required validation |
|---|---|---|
| Dev tooling (linters, formatters) | Low | CI passes |
| Test frameworks | Medium | All tests pass, no changed assertions |
| HTTP clients / SDKs (Azure, AWS) | High | Integration tests + semantic change review |
| Auth libraries | Critical | Security changelog review + penetration test |
| ORM / database clients | High | Migration test + data integrity verification |
| Core framework (Express, NestJS) | Critical | Full regression + canary deploy |

## 7. Common Pitfalls & Misconceptions

**"The tests pass, so the refactor is safe."**
Tests verify what they cover. A refactor that moves a side effect from before to after an await — changing whether it runs on success, failure, or both — will not be caught by tests that only assert the return value. Read the diff for side effect order changes, not just structure changes.

**"I'll ask Claude Code to refactor and fix the bug at the same time."**
Combined refactor+fix PRs are the hardest to review and the hardest to revert. If the fix introduces a regression, you can't cleanly revert just the fix. Always separate refactoring from logic changes — even if it means two consecutive PRs. The audit trail is worth it.

**"Claude Code read the CHANGELOG so the upgrade is thorough."**
Claude Code reads what it can access. If the migration guide is on a wiki page, in a GitHub release note, or buried in a discussion thread — and you didn't link it — Claude Code doesn't have it. Semantic changes in dependencies are frequently documented in places that aren't the `CHANGELOG.md`. Do your own read for critical dependencies.

**"The output audit takes too long — I'll just correct things as they come up."**
Reactive correction is O(n) — you correct the same mistake n times over n sessions. Audit-driven `CLAUDE.md` updates are O(1) — you correct it once and it applies forever. The 30-minute weekly audit pays back in the first session that runs clean because of a standing instruction.

**"If I add too many instructions to CLAUDE.md, Claude Code will get confused."**
Well-structured `CLAUDE.md` files with clear sections (conventions, forbidden patterns, environment, testing policy) scale to hundreds of lines without degrading output quality. Vague or contradictory instructions are the problem, not volume. Specific, actionable instructions compound positively.

## 8. Trade-offs, Scale, and Edge Cases

For very large refactors (decomposing a 2,000-line module into five), the risk of context drift across many file changes is real. Split into phases: Phase 1 extracts private methods within the same file. Phase 2 (a separate PR) moves those methods to new files. Each phase is independently reviewable and revertable.

Dependency upgrades across major version jumps (v5 → v7) often have multiple sequential migration guides. Do not ask Claude Code to apply all changes from v5 to v7 in one step. Apply v5 → v6 first, validate, then v6 → v7. Each intermediate version's breaking changes are documented separately for a reason.

The output audit is most valuable in the first 30–60 days of Claude Code adoption on a new codebase, when standing instructions are being discovered. After that, the cadence can drop to bi-weekly or be triggered by a pattern of consistent corrections. Don't audit for its own sake — audit when you notice you're correcting the same thing twice.

On Windows and CI environments: refactoring tools that involve shell commands (e.g., `grep` for string occurrences, running tests) may have path-separator or encoding differences. Always validate refactoring session shell outputs match what you'd see locally before closing a PR.

## 9. Self-Check Questions

1. You ask Claude Code to extract a 60-line block from a 300-line function into a private method. It does so. The tests pass. Before merging, you check one additional thing that tests can't catch. What is it and why?

2. You're upgrading `@azure/storage-blob` from v12.13 to v12.20. Claude Code updates all call sites. You run the tests — they pass. Is the upgrade complete? What's the one step Claude Code cannot do that you must do yourself?

3. You run your weekly output audit and notice Claude Code has been generating `try/catch` blocks that swallow errors without logging in three consecutive PRs. You approved all three. Write the standing instruction you'd add to `CLAUDE.md` to prevent the fourth.

4. A colleague argues that asking Claude Code to refactor and fix a bug at the same time is fine because "it's one PR either way." What's the concrete risk they're accepting, and how does it manifest when something goes wrong?

5. You've asked Claude Code to rename `PaymentService.charge()` to `PaymentService.processPayment()` across the codebase. It updates 23 call sites. What's the one category of usage that requires a separate manual check, and what command would you run to find it?

## 10. What to Learn Next

- **`CLAUDE.md` & Project Memory** — Every standing instruction discovered through this page's audit workflow lives in `CLAUDE.md`. Learning to structure it well — so instructions don't contradict each other and are easy to update — is what makes the compounding effect of audits real.
- **Hooks & Automation** — Manual post-refactor grep checks and post-upgrade test runs can be automated as hooks: run on every file write, report results immediately in the session. This page described the discipline; hooks enforce it without the discipline.
- **Multi-Agent Workflows** — Very large refactors (module-wide decompositions, codebase-wide dependency migrations) benefit from parallel agents: one per subsystem, one for tests, one for documentation. The per-PR review skills from this page apply to each agent's output independently.
- **Evaluation and Tracing for AI Systems** — The output audit described here is a manual evaluation loop. For teams using Claude Code at scale or building AI-integrated systems, formal evaluation frameworks provide structured metrics. This page is the intuition; that page is the infrastructure.

## Summary

Refactoring with Claude Code is safe when the scope is explicit, behaviour preservation is non-negotiable, and mechanical changes are separated from logical ones. Dependency upgrades require a three-phase approach: understand the breaking changes first (human + migration guide), update call sites (Claude Code), validate semantics (human + tests). Both workflows improve over time through the same mechanism: a structured weekly audit that converts reactive corrections into standing instructions in `CLAUDE.md`. The engineers who compound fastest with Claude Code are the ones who treat each correction as an investment — fixing the prompt, not just the code.
