# Claude Code: Feature Building & Multi-File Change Workflows
**Last Updated:** 2026-04-15

> *A feature isn't a file change — it's a decision about architecture, contracts, and future surface area. The engineers who ship well with Claude Code know how to describe a feature before they describe the code.*

---

## Learning Outcomes

By the end of this page, you will be able to:

- Structure a feature request with functional requirements, non-functional requirements, and architectural constraints so that Claude Code produces a plan worth reviewing
- Evaluate a Claude Code plan before approving it — knowing what to look for at each complexity tier
- Direct simple (single-file), intermediate (multi-file), and complex (new architectural component) feature changes with appropriate workflows for each
- Decompose a complex feature into a sequence of safe, reviewable steps rather than a single large instruction
- Design the change to be modular — so future requirements don't require rewriting the approach
- Apply this workflow to real Azure-adjacent scenarios (API extensions, service integrations, infrastructure additions)

---

## 1. Why This Matters (In Our Systems)

Two engineers are asked to add rate limiting to the public API.

Engineer A types: *"add rate limiting to the API."* Claude Code adds a middleware. It works. It's hardcoded to 100 requests per minute per IP. It lives in `app.ts` inline. Three months later, rate limiting needs to be configurable per-tenant, and the configuration needs to come from Azure App Configuration. The inline middleware is now a wall — everything needs to be rewritten because nothing was designed to be extended.

Engineer B spends 90 seconds writing a requirement that includes: the rate limiting strategy (sliding window, per-tenant), the configuration source (Azure App Configuration), the non-functional constraints (must not add more than 5ms to p99 latency), and the modularity requirement (the strategy should be swappable without touching the router). Claude Code produces a plan with a `RateLimitStrategy` interface, an `AppConfigRateLimitProvider`, and a middleware factory. When the requirements change, the extension point already exists.

The difference isn't the model. It's that Engineer B described the feature as a *design decision* before asking for an implementation.

## 2. Intuition & Mental Models

### The Three Complexity Tiers

Think of features on a spectrum of architectural impact:

**Tier 1 — Simple:** One logical change, one or two files, no new abstractions needed. *"Add a field to this API response."* The change fits in your head. Claude Code can execute in Auto mode with a focused instruction.

**Tier 2 — Intermediate:** Multiple files change together to implement a coherent behaviour. Controllers, services, tests, maybe config. *"Add an audit log for all user data mutations."* The risk here is consistency — if files drift out of sync, the feature breaks silently. Claude Code needs to understand the full surface before touching anything.

**Tier 3 — Complex:** The feature requires a new architectural component — a new interface, a new service layer, a new pattern that other code will depend on. *"Add a pluggable notification system that supports email, SMS, and webhook channels."* The risk here is lock-in — an implementation that makes the next requirement impossible without a rewrite.

**The mental model that cuts across all three:** before Claude Code writes a single line, it should produce a plan. Before you approve that plan, you should ask: *"if the next unknown requirement comes in, does this design accommodate it without rewriting the approach?"*

### Features as Contracts

Every feature you add becomes a contract — with the users who depend on it, with the tests that verify it, and with the next engineer who extends it. Thinking about the contract before the code is the difference between a feature that ages well and one that becomes technical debt in six months.

Claude Code is fast at implementation. It is not fast at knowing your future requirements. You are. The design conversation happens before the implementation, not during it.

## 3. Core Concepts & Terminology

**Feature brief** — The structured description of what a feature must do (functional requirements), how it must behave under constraints (non-functional requirements), what it must not do (exclusions), and how it will be extended in future (extensibility requirements). This is the input Claude Code needs to produce a plan worth reviewing.

**Plan-first workflow** — Running Claude Code in Plan mode for all Tier 2 and Tier 3 features. The plan is reviewed and approved before any file is written. This is non-negotiable for complex features.

**Decomposition** — Breaking a complex feature into a sequence of smaller, independently reviewable steps. *"First, introduce the interface. Second, implement the default provider. Third, wire it into the existing flow. Fourth, add tests. Fifth, add the secondary provider."* Each step is safe to merge on its own.

**Modularity constraint** — An explicit requirement in the feature brief that the implementation must not make future extensions harder. Expressed as: *"the X strategy must be swappable without modifying the router/controller/caller."*

**Extension point** — An interface, abstract class, factory, or configuration hook that allows new behaviour to be added without modifying existing code. Extension points are designed in, not retrofitted.

**Architectural component** — A new module, service, or layer that other code will depend on. Adding one requires answering: what does it own, what are its boundaries, what does it expose, and what is it forbidden from knowing about?

## 4. How It Works (What Actually Matters)

### How Claude Code Handles Multi-File Changes

When Claude Code executes a multi-file feature, it builds a dependency graph in its working context: which files need to change, in what order, and what each change depends on. It reads the files, understands their current structure, and plans edits that are consistent with each other.

The risk is **context drift** — when a session is long, earlier context can be implicitly deprioritised in favour of more recent instructions. In a 30-file change, the convention established in file 1 may not be consistently applied in file 28. This is why decomposition matters: smaller steps keep the context window focused.

> ⚠️ **Counterintuitive:** Asking Claude Code to make 30 file changes in one instruction is slower and riskier than asking it to make 5 coherent changes in 6 sequential instructions. The total wall-clock time is similar, but the reviewability and consistency are dramatically better.

### The Plan You Should Approve

A good plan from Claude Code for a Tier 2 or Tier 3 feature contains:

1. **A list of files that will change** — with a one-line reason for each
2. **A list of new files that will be created** — with a description of what each owns
3. **The interfaces or contracts that will be introduced** — before any implementation details
4. **What will explicitly NOT change** — the blast radius is bounded
5. **The order of changes** — so you can verify the sequence is safe

If a plan from Claude Code doesn't include items 3 and 5 for a complex feature, ask for them before approving.

> ⚠️ **Counterintuitive:** A plan that says "I'll update all the relevant files" is not a plan — it's a promise. A plan that says "I'll create `INotificationProvider.ts`, then implement `EmailNotificationProvider.ts`, then update `NotificationService.ts` to depend on the interface" is a plan you can evaluate.

## 5. Worked Examples & Realistic Scenarios

### Scenario 1: Tier 1 — Simple Feature (Single Concern)

**Feature:** Add a `correlationId` field to all API error responses in an Express/Node.js app.

**Instruction:**

```
Add a correlationId field to all API error responses in this Express app.

Functional:
- Every error response (4xx and 5xx) must include a correlationId field
- correlationId should be the value of the x-correlation-id request header,
  or a newly generated UUID if the header is absent
- Format: { "error": "...", "correlationId": "uuid-here" }

Non-functional:
- Do not change the HTTP status codes or existing error message format
- Must work with the existing error handler middleware in src/middleware/errorHandler.ts
- No new dependencies — use the crypto module for UUID generation

Files likely involved: src/middleware/errorHandler.ts
Test file: src/__tests__/middleware/errorHandler.test.ts
```

This is Tier 1: one file changes, the scope is unambiguous, Auto mode is appropriate. The instruction specifies the exact file, the exact format, the no-new-dependency constraint, and the test file location.

---

### Scenario 2: Tier 2 — Intermediate Feature (Multi-File, Consistent Behaviour)

**Feature:** Add audit logging for all user data mutations (create, update, delete) across the user management module.

**Instruction (Plan mode):**

```
I want to add audit logging for all user data mutations in the user
management module. Use Plan mode — do not write any code until I approve.

Functional requirements:
- Every create, update, and delete operation on a User entity must produce
  an audit log entry
- Audit log entry shape: { action, entityType, entityId, actorId, timestamp, diff }
- diff should capture what changed (before/after for updates)
- Audit logs must be written to the audit_logs table (schema in db/schema.sql)

Non-functional requirements:
- Audit logging must not be in the hot path — use async fire-and-forget,
  non-blocking. A logging failure must never fail the user operation.
- No audit logging in test environments (NODE_ENV === 'test')
- Must be consistent across all mutation operations — do not add it to
  controllers; add it in the service layer so it's always called regardless
  of the entry point

Modularity:
- The audit logging mechanism should be injectable/swappable — wrap it in
  an AuditLogger interface so we can swap the persistence layer later
  (e.g., move to Azure Event Hub) without changing service code

Files in scope: src/users/ directory only.
Do not touch: src/auth/, src/billing/, anything outside src/users/

Produce the plan first: list every file that changes, every file that
is created, the interfaces introduced, and the order of changes.
```

The modularity constraint is the most important line. It forces an `AuditLogger` interface into the plan — which means future changes (swap to Event Hub, add filtering, add replay) are extension points, not rewrites.

**What to look for in the plan Claude Code returns:**

- Is `AuditLogger` defined as an interface before any implementation?
- Is the concrete implementation (`DatabaseAuditLogger` or similar) separate from the interface?
- Is the service layer the injection point — not the controllers?
- Is the fire-and-forget pattern explicit in the plan (not just assumed)?
- Are tests included in the plan, not as an afterthought?

If any of these are missing, amend the plan before approving.

### Scenario 3: Tier 3 — Complex Feature (New Architectural Component)

**Feature:** Add a pluggable notification system supporting email (SendGrid), SMS (Azure Communication Services), and in-app notifications — with the ability to add new channels without modifying existing notification call sites.

This is a new architectural layer. Getting it wrong means three channels worth of hardcoded coupling. Getting it right means future channels are a new file and a registration.

**Two-phase approach:**

**Phase 1 — Design the contract first (Plan mode, no code):**

```
I want to design a pluggable notification system before writing any code.
Do not write any implementation. Produce only:

1. The interface definition for a NotificationProvider
2. The interface for NotificationPayload (channel-agnostic message)
3. The interface for NotificationRouter (routes a payload to one or
   more providers based on type/user preferences)
4. A description of how a new channel would be added in future —
   what files would a new developer need to create, and what
   existing files (if any) would need to change

Constraints:
- Adding a new channel must not require modifying NotificationRouter
  or any existing provider
- Providers must be registerable at startup — not hardcoded in the router
- The system must support sending to multiple channels for a single event
  (e.g., both email and in-app for an order confirmation)
- Existing call sites should call notificationService.send(payload)
  with no knowledge of which providers are active
```

Review the interfaces. Only when the contracts are right do you proceed to Phase 2.

**Phase 2 — Implement against the approved contracts:**

```
The interfaces are approved. Now implement:

Step 1: Create the interface files as defined in the plan.
  Files: src/notifications/INotificationProvider.ts,
         src/notifications/NotificationPayload.ts,
         src/notifications/NotificationRouter.ts
  Stop and show me the files before proceeding.

Step 2: Implement EmailNotificationProvider using SendGrid SDK
  (already installed: @sendgrid/mail).
  File: src/notifications/providers/EmailNotificationProvider.ts
  Include unit tests in src/notifications/providers/__tests__/

Step 3: Implement ACSNotificationProvider using Azure Communication
  Services SDK (@azure-communication-services/communication-email).
  File: src/notifications/providers/ACSNotificationProvider.ts
  Include unit tests.

Step 4: Implement InAppNotificationProvider — writes to notifications
  table in the database using existing db client in src/db/client.ts.
  File: src/notifications/providers/InAppNotificationProvider.ts
  Include unit tests.

Step 5: Wire the router into the app startup in src/app.ts.
  Register all three providers. Do not change any existing call sites —
  just make the router available via dependency injection.

Execute one step at a time. Stop after each step and wait for approval.
```

The "stop after each step" instruction is not caution theatre. It's the mechanism that keeps a 15-file change reviewable. Each step is a logical unit you can read, understand, and approve independently.

---

### Scenario 4: Modularity Test — Simulating the Unknown Future Requirement

Before accepting any Tier 3 feature plan, apply this test mentally:

*"The next requirement is: add a Microsoft Teams notification channel."*

If the answer is *"create `TeamsNotificationProvider.ts`, implement `INotificationProvider`, register it at startup — no other files change"* — the design is modular.

If the answer is *"update `NotificationRouter.ts`, add a case to the switch statement, update the config schema, update the tests..."* — the design has a coupling problem. Go back to Phase 1.

Ask Claude Code to run this test explicitly:

```
Before I approve this plan, simulate the following future requirement:
"Add a Microsoft Teams notification channel."

Describe exactly which files would be created, which existing files
would need to change, and why. If more than 2 existing files need
to change to add a new channel, explain what design decision caused
that coupling and propose how to fix it.
```

This question has caught more design problems than any code review.

## 6. Practical Usage & Decision Guidance

### Workflow by Complexity Tier

| Tier | Mode | Instruction style | Review point |
|---|---|---|---|
| Tier 1 (simple) | Auto or Accept Edits | Single instruction with file name + NFRs | `git diff` at end |
| Tier 2 (multi-file) | Plan then Auto | Full brief with modularity constraint | Review plan before executing |
| Tier 3 (architectural) | Plan (Phase 1: interfaces only) then step-by-step | Two phases: contracts first, implementation second | Approve interfaces before any implementation |

### Feature Brief Checklist

Before submitting any Tier 2 or Tier 3 instruction, verify:

- [ ] Functional requirements state what the feature *does*, not how
- [ ] Non-functional requirements state latency, reliability, security constraints
- [ ] Files in scope are named explicitly
- [ ] Files out of scope are named explicitly ("do not touch X")
- [ ] Modularity constraint is stated: *"X must be swappable without changing Y"*
- [ ] For Tier 3: interfaces are designed before implementation is requested
- [ ] For multi-step execution: each step ends with "stop and wait for approval"

### When to Decompose

Decompose into sequential steps when:
- More than 5 files change
- A new interface or contract is being introduced
- The feature has a natural "contract definition" phase separate from "implementation" phase
- The implementation will take more than one Claude Code session

Do NOT decompose when:
- The change is mechanical and consistent (e.g., add a field to 8 DTOs) — one instruction with all 8 files named is cleaner
- The steps have no natural review points — forced decomposition adds friction without safety

## 7. Common Pitfalls & Misconceptions

**"I described the feature, it produced code, and it looked right."**
Looking right is not the same as being right. A feature that works today but locks in the wrong abstraction is the most expensive kind of technical debt — it's invisible until the next requirement arrives. Always ask: *"if the next requirement is X, what breaks?"*

**"Plan mode showed me a list of files — that's the plan."**
A list of files is a scope estimate, not a plan. A plan shows the contracts that will be introduced, the order of changes, and what will explicitly not change. If Claude Code's plan doesn't include the interfaces and the change sequence, ask for them before approving.

**"I decomposed the feature into steps but ran all steps in one message."**
Decomposition only works if you stop between steps. The review point is the value — not the decomposition itself. Running all steps sequentially in Auto mode without pausing defeats the purpose.

**"I'll add the modularity constraint later if we need it."**
The modularity constraint is vastly cheaper to design in than to retrofit. A `NotificationRouter` without an interface costs one line to add before implementation. The same change after 3,000 lines of code means touching every caller, rewriting tests, and explaining to the team why the contract changed.

**"Complex features need complex instructions."**
The opposite is true. Complex features need *precise* instructions, which are often shorter than imprecise ones. *"The provider must implement INotificationProvider — nothing else"* is five words and eliminates an entire class of over-engineering.

## 8. Trade-offs, Scale, and Edge Cases

The two-phase approach (contracts then implementation) adds 10–15 minutes to a feature session. For Tier 1 and small Tier 2 features, this overhead isn't justified. Apply it selectively: when a new abstraction is being introduced, when the feature will be a dependency for other features, or when the team has been burned by a similar premature closure before.

For very large features (new microservice, full module rewrite), Claude Code alone is not the right tool for the initial architecture decision. Use it to validate a design you've already sketched, not to originate one from scratch. The model doesn't know your team's operational preferences, your Azure cost constraints, or your deployment topology. Your architecture sketch plus Claude Code's implementation is more reliable than Claude Code's architecture from scratch.

On Azure specifically: new architectural components often have infrastructure twins (a new Azure Function, a new Service Bus topic, a new App Configuration key prefix). The code-side and infra-side changes should be planned together. If your feature requires a new Azure resource, include that in Phase 1 — *"the notification system will write to an Azure Service Bus topic — include the Bicep resource definition in the contract phase."*

## 9. Self-Check Questions

1. You describe a feature to Claude Code and it produces a plan that lists 12 files that will change. Before approving, what three specific things do you look for in the plan that determine whether it's safe to execute?

2. You're building a feature that today needs to support two payment providers. You know a third will be added in six months. How does that future requirement change your instruction to Claude Code today?

3. Claude Code proposes implementing the notification router as a switch statement on channel type. You apply the "new channel" modularity test. What does the test reveal, and how do you ask Claude Code to revise the design?

4. A Tier 2 feature requires consistent changes across 8 DTO files, 8 service files, and 8 test files. Do you decompose this into steps, or write one instruction? What's your reasoning?

5. You're three steps into a decomposed feature implementation. The changes so far look correct. In step 4, Claude Code proposes an approach that contradicts a decision you made in step 1. What do you do, and what does this reveal about the instruction you gave at the start?

## 10. What to Learn Next

- **`CLAUDE.md` & Project Memory** — Feature briefs are powerful when written fresh, but the architectural constraints (patterns we follow, layers we respect, components that exist) belong in `CLAUDE.md` so Claude Code starts every session knowing the design landscape it's operating in.
- **Multi-Agent Workflows** — When a feature is large enough that sequential decomposition across a single session isn't practical, coordinating multiple parallel agents — one per subsystem — is the next level. The contract-first design pattern from this page is the prerequisite.
- **Claude Code Hooks & Automation** — After a feature is built, hooks can automate the validation loop: run tests after every step, check for interface compliance, enforce naming conventions without manual review.
- **Evaluation and Tracing for AI Systems** — If the feature you're building includes an AI component (a model call, a pipeline, an agent), the feature building workflow changes — model inputs and outputs become part of the contract, not implementation details.

## Summary

Feature building with Claude Code works at three tiers: simple changes (one concern, one file, Auto mode), intermediate changes (multi-file consistency, plan-first), and complex changes (new architectural components, contracts before implementation). The highest-leverage habit in all three tiers is the modularity constraint — stating explicitly that the new abstraction must be swappable without modifying its callers. This single requirement, added upfront, is the difference between a feature that ages well and one that becomes a rewrite three requirements later. For Tier 3 features, design the interfaces before asking for implementation, simulate the next unknown requirement before approving the plan, and decompose execution into reviewable steps. The model is fast; your design judgement is the constraint it can't replace.
