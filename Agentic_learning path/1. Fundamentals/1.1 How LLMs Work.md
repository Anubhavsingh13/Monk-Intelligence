## 1. Why This Matters (In Our Systems)

Imagine you ship a feature that uses an LLM to summarize customer feedback. It works perfectly in dev — warm, accurate, concise. Three weeks later in production, users start seeing summaries that confidently say the wrong thing. Support tickets pile up. You open the logs and see... the same prompt, the same model. What changed?

If you don't understand how LLMs work, you can't answer that question. You can't predict when they'll fail, you can't design for their constraints, and you'll spend your debugging time guessing instead of reasoning.

LLMs are not APIs that return the right answer. They're systems that return a *plausible continuation* of text. That distinction is everything.

---

## 2. Intuition & Mental Models

Here's the mental model that changes how you think about everything else:

**An LLM is an extremely good autocomplete.**

Not the weak kind that finishes your email subject line. The kind that has read most of the internet, every textbook it could find, and an enormous amount of code — and learned, at a statistical level, what text tends to follow what other text.

When you send a message, the model doesn't look up an answer. It asks: *given everything in this conversation so far, what text is most likely to come next?* Then it picks a word (actually a token). Then it asks again. And again. Until it decides to stop.

That's it. There's no reasoning engine, no database lookup, no planning module. Just an extraordinarily well-calibrated "what comes next" machine, applied one token at a time.

This explains a lot:
- Why it sounds confident when it's wrong — plausible and true are not the same thing
- Why adding examples to your prompt improves results — you're showing it what "next" looks like
- Why longer context sometimes helps and sometimes hurts — it's all just text being processed together

## 3. Core Concepts & Terminology

**Token**
Not a word — a chunk. "unbelievable" might be 3 tokens: `un`, `believ`, `able`. Code, JSON, and non-English text tend to tokenize less efficiently. This matters because pricing, context limits, and speed are all measured in tokens, not words. Roughly: 1 token ≈ 0.75 English words.

**Context Window**
The total amount of text (input + output) the model can "see" at once. Think of it as working memory. Older models had 4k tokens. Current models go up to 200k+. Beyond the window, the model has no memory of earlier conversation — it's not being evasive, it genuinely cannot see it.

**Temperature**
A dial between 0 and 1 (sometimes higher) that controls randomness. At 0, the model always picks the most likely next token. At 1, it samples more freely. For factual tasks: low temperature. For creative tasks: higher. For production pipelines that need consistency: closer to 0.

**System Prompt**
A special instruction block that comes before the user's message. This is where you define persona, constraints, output format, and context. It's the most powerful lever you have. Most reliability problems are system prompt problems.

**Embeddings**
Vector representations of text — numbers that capture meaning. Similar concepts end up numerically close. This is how semantic search, RAG, and similarity matching work. Embeddings are separate from generation but deeply related.

**RAG (Retrieval-Augmented Generation)**
A pattern where you fetch relevant documents from a database and inject them into the prompt before asking the model a question. The model doesn't "know" your internal docs — but if you paste the relevant ones into context, it can reason over them.

## 4. How It Works (What Actually Matters)

At inference time, here's the chain:

1. Your prompt is tokenized into a sequence of integers
2. Each token gets converted to an embedding vector (its "meaning" in high-dimensional space)
3. The transformer architecture processes the entire sequence — each token attending to every other token via the attention mechanism
4. The output is a probability distribution over the entire vocabulary (~100k tokens)
5. One token is sampled from that distribution
6. That token is appended to the sequence, and the whole thing repeats

Each forward pass through the model to produce one token requires significant compute. This is why latency scales with output length, not input length. Generating 1000 tokens takes roughly 10x longer than 100, regardless of how long your prompt was.

> **Counterintuitive:** The model doesn't read your prompt and then think. It processes everything — your instructions, the conversation history, its own previous output — as a single flat sequence, every time it generates a token. There's no separate "understanding" phase.

**Training** is separate from inference. The model was trained once (or a few times with fine-tuning) on massive text corpora using gradient descent. At inference, the weights are frozen. It is not learning from your conversations in real-time.

## 5. Worked Examples & Realistic Scenarios

**Scenario: Your summarizer starts hallucinating**

You have a C# service that calls the OpenAI API to summarize support tickets. It worked in testing. In production, some summaries mention features the ticket never discussed.

Here's how to reason through it:

First, is the input changing? If tickets grew longer and started hitting the context window limit, the model might be receiving truncated input — and filling the gaps with plausible-sounding content from training data.

Second, is temperature non-zero? If you set `temperature: 0.7` for "naturalness," you're allowing significant variance. For factual extraction tasks, use `temperature: 0` or `0.1`.

Third, is your system prompt explicit about staying grounded? Add: *"Only use information explicitly present in the ticket. If something is unclear, say so."* Constraints in the system prompt are not guaranteed, but they shift the probability distribution toward compliant outputs.

```python
response = client.chat.completions.create(
    model="claude-sonnet-4-6",
    temperature=0.1,  # low, for consistency
    system="Summarize only what is explicitly stated in the support ticket. Do not infer or add detail.",
    messages=[{"role": "user", "content": ticket_text}]
)
```

The fix isn't magic — it's understanding that the model generates *plausible* text, and your job is to make "accurate and grounded" more plausible than "hallucinated but fluent."

## 6. Practical Usage & Decision Guidance

| Situation | Recommendation |
|---|---|
| Extracting structured data from text | Low temperature (0–0.2), explicit output schema in system prompt, validate output |
| Creative generation or drafting | Temperature 0.6–0.9, looser constraints |
| User-facing chat | Streaming output, session-based context management |
| Working with private/internal docs | RAG pattern — don't fine-tune for knowledge, fine-tune for behavior |
| Needing consistent output across runs | Temperature 0, seed parameter if supported |
| Long documents that exceed context | Chunking + map-reduce or RAG, not just truncation |

**When NOT to use an LLM:**
- When you need deterministic outputs (use rule-based logic)
- When accuracy is life-critical without human review
- When latency is under 50ms (models are slow)
- When you have structured data and a defined schema (SQL or a typed parser will outperform)

## 7. Common Pitfalls & Misconceptions

**"It understood my instructions."**
The model doesn't understand anything. It produced output that statistically follows from your instructions. If the output is wrong, the model didn't "misunderstand" — the probability distribution for the response you wanted wasn't high enough. Adjust the prompt.

**"It was right yesterday."**
LLMs are non-deterministic at temperature > 0. The same prompt can produce different outputs on different runs. Build pipelines that validate outputs, not ones that assume consistency.

**"More context = better answers."**
Beyond a point, longer context hurts. Models lose track of details buried in the middle of very long contexts (the "lost in the middle" problem). Be selective about what you include.

**"Fine-tuning will fix hallucinations."**
Fine-tuning adjusts style and format. It does not reliably inject new factual knowledge and won't stop a model from confabulating. For knowledge, use RAG. For behavior, consider fine-tuning.

## 8. Trade-offs, Scale, and Edge Cases

**Latency vs. quality:** Larger models produce better outputs but with higher latency and cost. For user-facing features, a smaller model with a good prompt often beats a larger model with a lazy one.

**Context cost:** At scale, long system prompts and conversation histories become expensive fast. Every token in every request is billed. Cache static portions of prompts where your provider supports it.

**Rate limits:** LLM APIs have tokens-per-minute and requests-per-minute limits. At scale you need queuing, exponential backoff, and potentially multiple API keys or providers. Design for this from the start, not as an afterthought.

**Prompt injection:** When user input flows into a system prompt or is concatenated with trusted instructions, malicious users can override your instructions. Treat LLM inputs with the same suspicion as SQL inputs. Sanitize, structure, and isolate user content from system instructions.

> **Counterintuitive:** A model with a 200k token context window can still perform poorly on tasks requiring recall from the middle of that context. Bigger window is not a substitute for good retrieval design.

## Summary

LLMs are next-token prediction engines trained on vast text corpora — they generate plausible continuations, not looked-up answers. Understanding tokens, context windows, and temperature gives you the mental model to design prompts intentionally rather than by trial and error. At scale, the challenges shift from "will it produce good output" to "will it produce consistent, safe, observable output under load" — which requires validation, RAG patterns, and treating LLM calls with the same rigor as any external API call. The gap between a demo and a production LLM feature is mostly made of these fundamentals.
