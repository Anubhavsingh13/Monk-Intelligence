## 1. Why This Matters (In Our Systems)

Here's a scene that happens more than anyone admits: an engineer integrates an LLM into a feature, tests it with five examples, ships it — and then spends the next two weeks firefighting edge cases that "the model just got wrong."

The model didn't get it wrong. The prompt got it wrong. The model did exactly what it was most likely to do, given what it was told.

Prompt engineering is not a soft skill or a workaround. For any team shipping gen AI features, it is the primary interface between your intent and the model's output. Bad SQL makes a query slow. A bad prompt makes a feature unpredictable — and unpredictable in ways that are hard to reproduce, test, or explain to a stakeholder.

If you can write good prompts, you can ship reliable AI features. If you can't, you're shipping on hope.

---

## 2. Intuition & Mental Models

You've used ChatGPT. You know that how you phrase a question changes the answer you get. Prompt engineering is that intuition, made systematic and applied to code.

Here's the most useful mental model: **the model is an incredibly well-read intern on their first day.**

They're smart, they're eager, they've read everything — but they don't know your codebase, your conventions, your edge cases, or what "good" looks like for your specific context. If you say "summarize this," they'll give you their best guess at what a summary should be. If you say "summarize this in three bullet points, each under 15 words, for a non-technical executive," you'll get something you can actually use.

The model doesn't push back. It doesn't ask clarifying questions. It produces the most plausible output for the input you gave it. Your job is to make the *right output* the most plausible one.

A second mental model: **prompts are specifications, not requests.** When you write a function signature in C#, you're not hoping the compiler understands your vibe — you're being precise. Prompts deserve the same rigor.

---

## 3. Core Concepts & Terminology

**System Prompt**
The privileged instruction block that defines the model's role, constraints, and output format. It runs before any user message. This is your most powerful lever — treat it like a contract with the model.

**User Prompt**
The dynamic input — usually derived from user action or application data. Assume this is untrusted, like any user input.

**Zero-Shot Prompting**
Giving the model a task with no examples. Works well for common tasks the model has seen during training. Fast and cheap — start here.

**Few-Shot Prompting**
Providing 2–5 examples of input→output pairs before the real task. The single most reliable way to improve output format and quality without fine-tuning. Think of it as showing, not telling.

**Chain-of-Thought (CoT)**
Asking the model to reason step-by-step before giving a final answer. Adding "think through this step by step" to a prompt measurably improves accuracy on multi-step tasks. The model is more likely to arrive at the right answer if it "shows its work."

**Temperature**
Controls output randomness. `0` = deterministic, picks the most likely token every time. `1` = creative, samples more freely. For production pipelines: use low temperature. For brainstorming: higher is fine.

**Output Format Constraint**
Explicit instructions about how the response should be structured — JSON, bullet points, a specific schema. The more precisely you define the output, the easier it is to parse programmatically.

## 4. How It Works (What Actually Matters)

The model processes your entire prompt as a single sequence and generates the most statistically likely continuation. This means:

- **Order matters.** Instructions near the end of a prompt carry more weight than those buried in the middle. Put your most important constraints last.
- **Specificity shifts probability.** Vague instructions allow many valid interpretations. Specific instructions narrow the distribution toward what you want.
- **The model will fill gaps.** If you don't specify output format, the model will choose one — and it might be different each time.

The anatomy of a well-structured prompt for a production feature:

```
[Role / Persona]         → Who the model is in this context
[Task Definition]        → What it needs to do, precisely
[Context / Input]        → The data it's working with
[Constraints]            → What it must and must not do
[Output Format]          → Exactly how to structure the response
[Examples (if needed)]   → What good looks like
```

> **Counterintuitive:** Adding more instructions does not always improve results. Past a point, long prompts with many constraints create conflicts the model resolves unpredictably. Say what matters; cut the rest.

## 5. Worked Examples & Realistic Scenarios

**Scenario: Extracting structured data from support tickets**

Bad prompt:
```
Summarize this support ticket.
```

The model will produce a paragraph. You wanted JSON. You'll get a different paragraph length each time. You can't parse it reliably.

Better prompt:

```python
system_prompt = """
You are a support ticket classifier. Extract the following fields from the ticket text.

Return ONLY valid JSON matching this schema — no explanation, no markdown:
{
  "category": "billing | technical | account | other",
  "severity": "low | medium | high",
  "one_line_summary": "<max 15 words>"
}

If a field cannot be determined, use null.
"""

user_prompt = f"Ticket:\n{ticket_text}"
```

Why this works:
- Role sets context ("support ticket classifier" shifts probability toward classification language)
- Schema is explicit — no ambiguity about field names or valid values
- "ONLY valid JSON" and "no markdown" prevents the common wrapping in ```json ... ``` that breaks JSON parsers
- Null handling is specified — otherwise the model invents values

**Scenario: Few-shot for tone-sensitive output**

If you need the model to match a specific writing style, describe it or show it:

```python
system_prompt = """
You write customer-facing error messages. They are calm, specific, and never blame the user.

Examples:
Input: "User's payment failed due to expired card"
Output: "We weren't able to process your payment. Please check that your card details are up to date and try again."

Input: "API rate limit exceeded"
Output: "You've hit the request limit for this period. Your access will reset at the top of the hour."

Now write a message for the following error:
"""
```

Two examples are usually enough to lock in tone and pattern. Three to five if the pattern is complex.

## 6. Practical Usage & Decision Guidance

| Task Type | Recommended Approach |
|---|---|
| Simple classification or extraction | Zero-shot with explicit output schema |
| Format-sensitive or stylistically specific output | Few-shot (2–5 examples) |
| Multi-step reasoning or complex decisions | Chain-of-thought (`"think step by step"`) |
| User-facing generated content | System prompt with constraints + low temperature |
| Batch processing pipeline | Low temperature (0–0.1), strict schema, output validation |
| Exploratory / brainstorming | Higher temperature, lighter constraints |

**When to add examples vs. when to improve instructions:**
If your zero-shot prompt is producing the right *kind* of output but not reliably, add examples. If it's producing the wrong *kind* entirely, fix the instructions first — examples won't rescue a broken specification.

## 7. Common Pitfalls & Misconceptions

**"I'll just tell it not to do the bad thing."**
Negative instructions ("do not include markdown", "never make up information") are weaker than positive ones. Instead of "don't use bullet points," say "respond in a single paragraph." Tell the model what to do, not just what to avoid.

**"The prompt worked in testing, so it's done."**
A prompt tested on 10 examples is not a production prompt. Edge cases in real data will find gaps in your specification you didn't know existed. Treat prompts like code: they need coverage.

**"More detail always helps."**
A 2000-word system prompt for a simple task is a liability. Irrelevant instructions add noise and can create contradictions the model resolves in unexpected ways. Minimum viable specificity is the goal.

**"I'll fix it with fine-tuning."**
If your prompt isn't working, fine-tuning is almost never the right first answer. It's expensive, slow, and doesn't fix a poorly specified task. Fix the prompt first. Fine-tune for behavior after the prompt is solid.

## 8. Trade-offs, Scale, and Edge Cases

**Cost:** Every token in your system prompt is billed on every request. A 500-token system prompt on 1 million daily requests is 500 million tokens per day just in overhead. At scale, prompt length is a cost line item. Audit it like one.

**Latency:** Longer prompts increase time-to-first-token. If you're streaming responses to a UI, this is the delay the user feels before anything appears. Keep prompts lean for latency-sensitive paths.

**Prompt injection:** When user input is interpolated into a system prompt or concatenated with trusted instructions, a malicious user can override your constraints. Always isolate user content from the instruction layer:

```python
# Bad — user input can escape the task boundary
prompt = f"Summarize this feedback: {user_input}"

# Better — structural separation
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_input}  # isolated
]
```

**Version your prompts.** Prompts change. Model behavior changes when providers update versions. Without version control and logging of which prompt produced which output, debugging production regressions becomes archaeology. Store prompt versions alongside model version and log both with every inference call.

**Eval before you ship.** At scale, "it looks right" is not a QA process. Build a small golden dataset — 20 to 50 representative inputs with expected outputs — and run your prompt against it before every change. This is not ML research; it's basic engineering discipline applied to a new kind of function.

## 9. Self-Check Questions

1. Your extraction prompt works 95% of the time but occasionally returns markdown-wrapped JSON that breaks your parser. What's the root cause and how do you fix it at the prompt level?
2. A colleague suggests adding 10 more constraints to a system prompt that's already producing inconsistent results. What do you ask them first?
3. You're building a batch pipeline that processes 500k documents per day. Your current system prompt is 800 tokens. What questions do you ask before shipping?
4. How would you design a test suite for a prompt — what goes in it, and how do you know when a prompt change is safe to deploy?
5. A user reports that the model "ignored" your system prompt instructions. What are three possible explanations and how would you investigate each?

## Summary

Prompt engineering is the discipline of writing instructions precise enough that the model's most likely output is the one you actually want. The core skills — specifying output format, using few-shot examples, constraining with a system prompt, and choosing the right temperature — are learnable and directly map to shipping more reliable AI features. At scale, prompts become cost, latency, and security concerns, not just quality ones: version them, test them, and treat them with the same rigor as the code that calls them. The gap between a demo that works and a feature that ships is mostly a well-engineered prompt.
